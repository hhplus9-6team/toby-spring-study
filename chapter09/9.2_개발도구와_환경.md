# 9.2 개발도구와 환경

## 서론

엔터프라이즈 애플리케이션 개발은 단순히 코드를 작성하는 것만으로 끝나지 않습니다.

대규모 프로젝트에서는 수많은 라이브러리와 복잡한 의존관계를 관리해야 하고, 이를 효율적으로 다루기 위한 개발 도구와 환경이 필수적입니다.

특히 스프링은 다양한 기술을 조합해 사용하는 프레임워크이기 때문에 개발 환경과 도구 선택이 생산성과 직결됩니다.

## 9.2.1 JavaSE와 JavaEE

### 1) Java SE / SDK

Java SE는 자바의 기본 실행 환경으로, 우리가 흔히 사용하는 “기본 자바”를 의미합니다.

- JVM
- 기본 라이브러리
- 컴파일러 (javac)

### 2) Java EE / J2EE

Java EE는 엔터프라이즈 개발을 위한 API 집합으로, 서버 개발용 기능 모음입니다.

- Servlet
- JSP
- JPA
- EJB

## 9.2.2 IDE

IDE는 단순한 코드 편집기가 아니라 개발 생산성을 극대화하는 도구입니다.

대표적으로 IntelliJ, Eclipse 등이 있으며, 다음과 같은 기능을 제공합니다.

- 코드 자동완성
- 디버깅
- 리팩토링
- 프로젝트 관리

스프링 같은 프레임워크는 설정과 구조가 복잡하기 때문에 IDE 지원 없으면 생산성 급감합니다.

## 9.2.3 SpringSource Tool Suite (STS)


TS는 스프링 개발을 지원하기 위해 만들어진 Eclipse 기반 IDE입니다.

과거에는 XML 기반 설정이 많았기 때문에 이를 도와주는 플러그인이 중요했습니다.

예를 들어 Bean 설정을 작성할 때 자동완성을 제공할 수 있습니다.

```xml
<bean id="userService" class="com.example.UserService"/>
```

현재는 IntelliJ가 대부분의 기능을 제공하기 때문에 STS의 사용은 줄었지만, 당시에는 스프링 개발 생산성을 크게 높여주는 도구였습니다.

## 9.2.4 라이브러리 관리와 빌드 툴

### 1) 라이브러리 관리의 어려움

자바 애플리케이션 개발에서는 다양한 외부 라이브러리를 함께 사용합니다.

하지만 초기 자바 개발 환경에서는 라이브러리를 매우 원시적인 방식으로 관리해야 했습니다.

```
lib/
 ├─ spring-context.jar
 ├─ spring-core.jar
 └─ commons-logging.jar
```

라이브러리를 직접 다운로드하여 프로젝트에 추가하고, 컴파일 시 classpath에 포함시키는 방식이었습니다.

```xml
<javac srcdir="src" destdir="bin" classpath="lib/*"/>
```

이 방식은 간단해 보이지만, 실제로는 다음과 같은 문제를 야기합니다.

- 어떤 라이브러리가 필요한지 직접 찾아야 한다
- 의존 라이브러리(전이 의존성)를 알기 어렵다
- 버전 충돌이 쉽게 발생한다

```
A 라이브러리 → log4j 1.x 필요
B 라이브러리 → log4j 2.x 필요
```

이 경우 어떤 버전을 선택해야 할지 개발자가 직접 판단해야 합니다.

### 2) 라이브러리 선정

라이브러리를 선택하는 것 또한 중요한 문제입니다.

- 기능
- 안정성
- 커뮤니티 지원
- 의존성 구조

특히 스프링처럼 다양한 모듈을 사용하는 경우에는 어떤 모듈을 선택하고 조합할 것인지도 중요한 설계 요소가 됩니다.

### 3) 빌드 툴과 라이브러리 관리

이 문제를 해결하기 위해 빌드 도구가 등장했습니다.

#### ANT: 초기 빌드 도구

ANT는 자바 프로젝트를 빌드하기 위한 초기 도구입니다.

```xml
<target name="compile">
    <javac srcdir="src" destdir="bin"/>
</target>
```
하지만 ANT는 단순히 빌드 과정을 자동화할 뿐, 라이브러리 의존성 관리는 지원하지 않습니다.

즉, jar 파일은 여전히 직접 관리해야 합니다.

#### Maven: 의존성 관리의 시작

Maven은 빌드와 함께 의존성 관리 기능을 도입했습니다.

```xml
<dependency>
    <groupId>org.springframework</groupId>
    <artifactId>spring-context</artifactId>
</dependency>
```

이 선언만으로 필요한 라이브러리와 그 의존성까지 자동으로 다운로드됩니다.

#### Gradle: 현재의 표준

Gradle은 Maven을 발전시킨 형태로, 현재 가장 많이 사용되는 빌드 도구입니다.

```groovy
dependencies {
    implementation 'org.springframework:spring-context'
}
```


#### (+) 의존성 충돌 해결

빌드 도구는 내부적으로 의존성 트리를 분석해 충돌을 자동으로 해결합니다.

```
여러 버전 → 하나 선택 (우선순위 규칙)
```

필요 시 라이브러리 패키지 이름을 변경하는 Shading(재패키징)으로 해결하기도 합니다.

### 4) 스프링 모듈의 두가지 이름

#### 모듈

예를 들어 스프링은 다음과 같은 모듈로 구성되어 있습니다.

```
spring-core
spring-context
spring-web
```

각 모듈은 특정 역할을 담당합니다.

- spring-core → 기본 기능
- spring-context → DI / 컨테이너
- spring-web → 웹 기능

즉, 스프링에서 말하는 모듈은 하나의 기능 단위로 나뉜 라이브러리를 의미합니다.

#### 자바에서 모듈 개념이 부족했던 이유

자바는 오랫동안 언어 차원에서 모듈 개념이 부족했습니다.

- 의존성 관리 없음
- 모듈 간 관계 관리 없음

그래서 과거에는 라이브러리를 다음과 같이 직접 관리해야 했습니다.

```
spring-core → 기본 기능
spring-context → DI / 컨테이너
spring-web → 웹 기능
```

어떤 모듈이 필요한지, 어떤 순서로 의존하는지를 전부 개발자가 직접 알아야 했습니다.

#### 멀티모듈

```
project
 ├─ api
 ├─ service
 └─ common
```

자바 기능이 아니라 Gradle / Maven 이 제공하는 기능입니다.

- 자바 → 모듈 개념 약함
- 빌드툴 → 모듈 구조 제공

#### 스프링 모듈의 두 가지 이름

1) artifactId (빌드 관점)

```
spring-context
```

Gradle / Maven 에서 사용합니다.

```groovy
implementation 'org.springframework:spring-context'
```
   
2) package (코드 관점)

```
org.springframework.context
```

코드에서 import 할 때 사용합니다.

```java
import org.springframework.context.ApplicationContext;
```

### 5) 리포지토리

스프링 모듈은 중앙 저장소(Repository)를 통해 관리됩니다.

대표적으로 Maven Central이 있습니다.

빌드 도구는 이 리포지토리에서 필요한 라이브러리를 자동으로 다운로드한다.

```
개발자 → dependency 선언 → 빌드툴 → 리포지토리 조회 → 다운로드
```
