# 9.1 자바 엔터프라이즈 플랫폼과 스프링 애플리케이션

스프링으로 만들 수 있는 애플리케이션의 종류에는 제한이 없다. 자바 언어를 사용하는 모든 종류의 프로젝트에 사용할 수 있다. 하지만 스프링이 가장 많이 사용되는 대표적인 영역은 **자바 엔터프라이즈 플랫폼(JavaEE)** 기반의 애플리케이션이다.

자바 엔터프라이즈 애플리케이션은 서버에서 동작하며 클라이언트를 상대로 서비스를 제공한다. 클라이언트의 요청을 받아서 그에 대한 작업을 수행하고 결과를 돌려주는 것이 기본적인 동작 방식이며, 정해진 시간이나 특정 이벤트 발생 시 독자적 작업을 수행하기도 한다.


## 9.1.1 클라이언트와 백엔드 시스템

엔터프라이즈 애플리케이션은 보통 **클라이언트** - **서버(애플리케이션)** - **백엔드** 세 가지 영역으로 나뉜다.

### 클라이언트

- 사용자 또는 외부 시스템으로부터 요청을 받아들이는 쪽
- **웹 브라우저**: 가장 흔한 클라이언트. HTTP를 통해 서버와 통신한다.
- **웹 클라이언트(리치 클라이언트)**: 플렉스(Flex)나 Ajax를 활용한 클라이언트. 서버와 데이터만 주고받으며 화면 구성은 클라이언트 쪽에서 처리한다.
- **다른 서버 애플리케이션**: 원격 통신을 통해 다른 서버가 클라이언트 역할을 할 수 있다.

### 백엔드 시스템

- 서버 애플리케이션이 사용하는 **데이터를 저장하고 관리하는 시스템**
- **DB**: 가장 많이 사용되는 백엔드 시스템. JDBC를 통해 연결한다.
- **레거시 시스템**: 기존 메인프레임에서 동작하는 시스템. JCA(Java Connector Architecture) 등을 통해 연결한다.
- **다른 서버 애플리케이션**: 다른 WAS에서 동작하는 애플리케이션과 연동할 수 있다.


## 9.1.2 애플리케이션 서버

스프링으로 만든 애플리케이션을 자바 서버환경에 배포하려면 **JavaEE(J2EE) 서버**가 필요하다. JavaEE 표준을 따르는 애플리케이션 서버는 크게 두 가지로 나뉜다.

### 경량급 WAS / 서블릿 컨테이너

- **아파치 톰캣(Tomcat)**, **제티(Jetty)** 등이 대표적이다.
- 서블릿 컨테이너(서블릿/JSP)만 지원하며, EJB나 리소스 커넥터 등의 JavaEE의 모든 기능을 지원하지는 않는다.
- 스프링은 EJB 같은 무거운 기술에 의존하지 않기 때문에, **서블릿 컨테이너만으로도 충분히 엔터프라이즈 애플리케이션을 개발**할 수 있다.
- 가볍고 빠르며 개발과 배포가 간편하다.

### 완전한 JavaEE 호환 애플리케이션 서버

- **WebLogic**, **WebSphere**, **JBoss** 등이 대표적이다.
- EJB, JCA, JMS 등 JavaEE의 모든 표준 기술을 지원한다.
- 서블릿 컨테이너만으로 충분한 경우가 많지만, 다음과 같은 경우에는 완전한 JavaEE 서버가 필요하다:
    - **분산 트랜잭션(JTA)**: 두 개 이상의 DB 또는 JMS 등에 대한 글로벌 트랜잭션이 필요한 경우
    - **레거시 시스템과의 연동(JCA)**: 메인프레임 등과 커넥터를 통해 연결해야 하는 경우
    - **기존 EJB 시스템과의 통합**: 이미 EJB로 개발된 모듈이 있는 경우

### 스프링소스 tcServer

- 아파치 톰캣을 기반으로, 스프링소스(SpringSource)에서 **스프링 애플리케이션에 최적화**하여 만든 경량급 WAS이다.
- STS(Spring Tool Suite)에 내장되어 있으며, 스프링 애플리케이션 개발 시 기본 서버로 사용할 수 있다.

> 스프링의 장점은 **서버환경에 대한 의존도가 낮다**는 것이다. 톰캣 같은 가벼운 서블릿 컨테이너에서도, WebLogic 같은 고가의 상용 WAS에서도 동일한 코드가 동작한다.


## 9.1.3 스프링 애플리케이션의 배포 단위

스프링으로 개발한 애플리케이션은 다양한 형태로 배포할 수 있다.

### 독립 웹 모듈 (WAR)

- **가장 일반적이고 보편적인 배포 형태**
- war로 패키징된 독립 웹 모듈로 배포한다.
- 웹 애플리케이션을 통째로 묶어서 서블릿 컨테이너나 JavaEE 서버에 배포한다.
- 톰캣 같은 서블릿 컨테이너를 쓴다면 독립 웹 모듈이 유일한 방법이다.
- 구성 요소:
    - **WEB-INF/web.xml**: 웹 애플리케이션의 배포 설명자(deployment descriptor). 서블릿, 필터, 리스너 등을 정의한다.
    - **WEB-INF/lib**: 라이브러리 jar 파일
    - **WEB-INF/classes**: 컴파일된 클래스 파일, 설정 파일
    - **기타 정적 리소스**: HTML, CSS, JS, 이미지 등

### 엔터프라이즈 애플리케이션 (EAR)

- ear로 패키징하여 **하나 이상의 웹 모듈(WAR)과 별도로 분리된 스프링 컨텍스트를 엔터프라이즈 애플리케이션으로 묶어** JavaEE 애플리케이션 서버에 배포한다.
- EJB 모듈을 포함해야 하거나, 여러 WAR를 하나의 애플리케이션 단위로 관리해야 할 때 사용한다.
- 완전한 JavaEE 서버에서만 배포 가능하다.

### 백그라운드 서비스 모듈 (RAR)

- rar로 **리소스 커넥터를 만들어** 배포하는 방식이다.
- UI를 가질 필요 없이 **백그라운드 서비스처럼** 동작하는 경우에 해당한다.
- 완전한 JavaEE 서버 환경에서 사용 가능하다.


> 특별한 이유가 없다면 **스프링 애플리케이션은 WAR로 패키징해서 서블릿 컨테이너에 올리는 것**이 가장 간편하고 보편적인 방법이다.


## 정리

- 스프링은 자바 엔터프라이즈 환경 전반에서 사용할 수 있으며, 특히 서블릿 기반 웹 애플리케이션에서 가장 많이 활용된다.
- 스프링은 EJB 같은 무거운 기술 없이도 **서블릿 컨테이너(톰캣 등)만으로 엔터프라이즈급 기능**을 제공할 수 있게 해준다.
- 배포 단위는 대부분 **WAR** 형태이며, 필요에 따라 EAR이나 RAR을 선택할 수 있다.
- 서버 환경에 대한 종속성이 낮아 **가벼운 서블릿 컨테이너에서 고가의 WAS까지** 동일한 코드로 동작한다는 것이 스프링의 큰 장점이다.
