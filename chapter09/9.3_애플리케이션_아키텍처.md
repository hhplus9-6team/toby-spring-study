# 9.3 애플리케이션 아키텍처

## 애플리케이션 아키텍처란?

애플리케이션 아키텍처는 단순히 폴더 구조를 나누는 것이 아니라, 애플리케이션의 책임을 어떻게 분리하고, 각 책임이 어떤 방식으로 협력하게 할 것인지를 정하는 구조입니다.

스프링 애플리케이션에서는 보통 다음과 같은 계층 구조를 많이 사용합니다.

```
Controller
↓
Service
↓
Repository
↓
Database
```

이 구조의 핵심은 관심사의 분리, 즉 SoC입니다.

Controller는 HTTP 요청과 응답을 다루고, Service는 애플리케이션의 비즈니스 흐름을 처리하며, Repository는 데이터 접근을 담당합니다.

## 9.3.1 계층형 아키텍처

### 아키텍처와 SoC

관심사의 분리란 각 코드가 자신이 맡은 역할에만 집중하게 만드는 것입니다.

예를 들어 아래 코드는 Controller가 너무 많은 일을 하고 있습니다.

```java
@RestController
@RequiredArgsConstructor
public class OrderController {

    private final OrderMapper orderMapper;
    private final CouponMapper couponMapper;

    @PostMapping("/orders")
    public ResponseEntity<Void> order(@RequestBody OrderRequest request) {
        Member member = orderMapper.findMember(request.getMemberId());

        int totalPrice = 0;
        for (OrderItemRequest item : request.getItems()) {
            Product product = orderMapper.findProduct(item.getProductId());
            totalPrice += product.getPrice() * item.getQuantity();
        }

        Coupon coupon = couponMapper.findCoupon(request.getCouponId());
        int discountPrice = coupon.calculateDiscount(totalPrice);

        int finalPrice = totalPrice - discountPrice;

        orderMapper.insertOrder(member.getId(), finalPrice);

        return ResponseEntity.ok().build();
    }
}
```

이 코드는 동작할 수는 있지만 문제가 많습니다.

Controller가 HTTP 요청 처리뿐 아니라 회원 조회, 상품 조회, 가격 계산, 쿠폰 할인, 주문 저장까지 모두 처리하고 있습니다. 이렇게 되면 Controller는 웹 계층의 역할을 넘어 비즈니스 로직과 데이터 접근까지 담당하게 됩니다.

이를 계층형 구조로 분리하면 다음과 같습니다.

```java
@RestController
@RequiredArgsConstructor
public class OrderController {

    private final OrderService orderService;

    @PostMapping("/orders")
    public ResponseEntity<Void> order(@RequestBody OrderRequest request) {
        orderService.order(request.toCommand());
        return ResponseEntity.ok().build();
    }
}
```
```java
@Service
@RequiredArgsConstructor
public class OrderService {

    private final ProductRepository productRepository;
    private final CouponRepository couponRepository;
    private final OrderRepository orderRepository;

    @Transactional
    public void order(OrderCommand command) {
        List<Product> products = productRepository.findAllByIds(command.productIds());
        Coupon coupon = couponRepository.findById(command.couponId());

        Order order = Order.create(command.memberId(), products, coupon);

        orderRepository.save(order);
    }
}
```

Controller는 요청을 받고 Service를 호출하는 역할만 합니다.
Service는 주문이라는 애플리케이션 흐름을 담당합니다.
Repository는 데이터 조회와 저장을 담당합니다.

이렇게 각 계층이 자신의 책임만 가지면 변경이 쉬워집니다.

### 3계층 아키텍처와 수직 계층

일반적인 3계층 구조는 다음과 같습니다.

```
프레젠테이션 계층
- Controller
- View
- API Request/Response 처리

서비스 계층
- 비즈니스 흐름 처리
- 트랜잭션 경계
- 여러 Repository, 외부 API 조합

데이터 액세스 계층
- Repository
- DAO
- Mapper
- DB 접근
```

각 계층은 아래 방향으로만 의존해야 합니다.

```
Controller → Service → Repository
```

반대로 Repository가 Service를 알거나, Service가 Controller를 알아서는 안 됩니다.

예를 들어 Service에서 HttpServletRequest를 직접 사용하는 코드는 좋지 않습니다.

```java
@Service
public class OrderService {

    public void order(HttpServletRequest request) {
        String userId = request.getHeader("userId");

        // 주문 처리
    }
}

```

Service가 웹 환경에 의존하게 되면, 나중에 배치, 메시지 큐, 테스트 코드 등 다른 실행 환경에서 재사용하기 어려워집니다.

더 나은 방식은 Controller에서 웹 요청을 애플리케이션이 이해할 수 있는 값으로 변환한 뒤 Service에 전달하는 것입니다.

```java
@RestController
@RequiredArgsConstructor
public class OrderController {

    private final OrderService orderService;

    @PostMapping("/orders")
    public ResponseEntity<Void> order(
            @RequestHeader Long userId,
            @RequestBody OrderRequest request
    ) {
        orderService.order(request.toCommand(userId));
        return ResponseEntity.ok().build();
    }
}
```

```java
@Service
public class OrderService {

    public void order(OrderCommand command) {
        Long userId = command.userId();

        // 주문 처리
    }
}
```

### 계층형 아키텍처 설계의 원칙

- 계층 간 역할을 명확히 분리합니다.
- 하위 계층에만 의존합니다.
- 계층 내부는 높은 응집도를 유지하고, 계층 간 결합도는 낮춥니다.

## 9.3.2 애플리케이션 정보 아키텍처 

애플리케이션에서 정보를 어떻게 다루는지에 따라 구조가 달라진다고 설명합니다.

크게 보면 다음 방식들이 있습니다.

1. DB/SQL 중심 방식
2. 거대한 서비스 계층 방식
3. 오브젝트 중심 방식

### DB/SQL 중심의 로직 구현 방식 

DB/SQL 중심 방식은 비즈니스 로직 대부분이 SQL에 들어가는 구조입니다.

```sql
SELECT
o.order_id,
SUM(oi.price * oi.quantity) AS total_price,
CASE
WHEN c.discount_type = 'RATE'
THEN SUM(oi.price * oi.quantity) * c.discount_rate / 100
WHEN c.discount_type = 'AMOUNT'
THEN c.discount_amount
ELSE 0
END AS discount_price
FROM orders o
JOIN order_item oi ON o.order_id = oi.order_id
LEFT JOIN coupon c ON o.coupon_id = c.coupon_id
WHERE o.order_id = #{orderId}
GROUP BY o.order_id, c.discount_type, c.discount_rate, c.discount_amount;
```

이 방식은 조회 성능을 내기 쉽고, 단순한 CRUD에서는 빠르게 개발할 수 있습니다.

하지만 비즈니스 규칙이 SQL에 숨어버립니다.

예를 들어 “정률 쿠폰은 최대 10,000원까지만 할인 가능하다” 같은 규칙이 생기면 SQL은 점점 복잡해집니다.

```sql
CASE
WHEN c.discount_type = 'RATE'
THEN LEAST(SUM(oi.price * oi.quantity) * c.discount_rate / 100, 10000)
WHEN c.discount_type = 'AMOUNT'
THEN c.discount_amount
ELSE 0
END AS discount_price
```

처음에는 괜찮아 보여도, 조건이 늘어날수록 SQL이 비즈니스 로직의 중심이 됩니다.
결과적으로 Java 코드는 단순히 SQL 결과를 전달하는 역할만 하게 됩니다.

```java
public OrderSummaryDto getOrderSummary(Long orderId) {
    return orderMapper.selectOrderSummary(orderId);
}
```

이 구조에서는 비즈니스 규칙을 테스트하기 어렵습니다.
SQL을 실행해야만 로직을 검증할 수 있기 때문입니다.

### 거대한 서비스 계층 방식

DB 중심 방식의 반대처럼 보이지만, 실무에서 흔히 나오는 구조가 “거대한 서비스 계층”입니다.

객체는 단순히 데이터를 담는 용도로만 쓰고, 모든 로직을 Service에서 처리합니다.

```java
@Service
@RequiredArgsConstructor
public class OrderService {

    private final ProductRepository productRepository;
    private final CouponRepository couponRepository;
    private final OrderRepository orderRepository;

    @Transactional
    public void order(OrderCommand command) {
        List<Product> products = productRepository.findAllByIds(command.productIds());

        int totalPrice = 0;
        for (Product product : products) {
            totalPrice += product.getPrice() * command.getQuantity(product.getId());
        }

        Coupon coupon = couponRepository.findById(command.couponId());

        int discountPrice = 0;
        if (coupon.getType() == CouponType.RATE) {
            discountPrice = totalPrice * coupon.getRate() / 100;
            if (discountPrice > coupon.getMaxDiscountAmount()) {
                discountPrice = coupon.getMaxDiscountAmount();
            }
        } else if (coupon.getType() == CouponType.AMOUNT) {
            discountPrice = coupon.getDiscountAmount();
        }

        int finalPrice = totalPrice - discountPrice;

        if (finalPrice < 0) {
            finalPrice = 0;
        }

        Order order = new Order(command.memberId(), totalPrice, discountPrice, finalPrice);

        orderRepository.save(order);
    }
}
```

이 방식은 SQL 중심 방식보다 Java 코드에서 로직을 볼 수 있다는 장점이 있습니다.
하지만 시간이 지나면 Service가 너무 커집니다.

Service가 다음 책임을 모두 갖게 됩니다.

- 상품 조회
- 주문 금액 계산
- 쿠폰 정책 판단
- 할인 금액 계산
- 최종 결제 금액 계산
- 주문 생성
- 주문 저장

결국 Service가 모든 것을 아는 객체가 됩니다.

이 구조에서는 도메인 객체가 사실상 DTO처럼 사용됩니다.

```java
public class Order {
    private Long memberId;
    private int totalPrice;
    private int discountPrice;
    private int finalPrice;
    
    // getter/setter
}
```

이런 객체를 “빈약한 도메인 오브젝트”라고 볼 수 있습니다.

## 9.3.3 오브젝트 중심 아키텍처 

오브젝트 중심 아키텍처는 데이터를 단순히 전달하는 것이 아니라, 데이터와 그 데이터를 다루는 로직을 함께 객체 안에 두는 방식입니다.

즉, 객체가 자기 책임을 가집니다.

### 데이터와 오브젝트 

데이터 중심 코드는 보통 이런 형태입니다.

```java
public class Coupon {
    private CouponType type;
    private int rate;
    private int discountAmount;
    private int maxDiscountAmount;

    public CouponType getType() {
        return type;
    }

    public int getRate() {
        return rate;
    }

    public int getDiscountAmount() {
        return discountAmount;
    }

    public int getMaxDiscountAmount() {
        return maxDiscountAmount;
    }
}
```

이 객체는 값을 가지고 있지만, 스스로 할 수 있는 일이 없습니다.
할인 계산은 외부 Service가 합니다.

오브젝트 중심으로 바꾸면 Coupon이 스스로 할인 금액을 계산합니다.

```java
public class Coupon {

    private CouponType type;
    private int rate;
    private int discountAmount;
    private int maxDiscountAmount;

    public int calculateDiscount(int totalPrice) {
        if (type == CouponType.RATE) {
            int discount = totalPrice * rate / 100;
            return Math.min(discount, maxDiscountAmount);
        }

        if (type == CouponType.AMOUNT) {
            return Math.min(discountAmount, totalPrice);
        }

        return 0;
    }
}
```

이제 Service는 쿠폰 타입이 정률인지 정액인지 몰라도 됩니다.

```java
int discountPrice = coupon.calculateDiscount(totalPrice);
```

이게 핵심입니다.

비즈니스 규칙을 가장 잘 아는 객체가 그 규칙을 책임지게 만듭니다.

### 도메인 오브젝트를 사용하는 코드 

Order도 마찬가지입니다.

```java
public class Order {

    private Long memberId;
    private List<OrderItem> items = new ArrayList<>();
    private int totalPrice;
    private int discountPrice;
    private int finalPrice;

    public static Order create(Long memberId, List<OrderItem> items, Coupon coupon) {
        Order order = new Order(memberId, items);
        order.calculateTotalPrice();
        order.applyCoupon(coupon);
        return order;
    }

    private Order(Long memberId, List<OrderItem> items) {
        this.memberId = memberId;
        this.items = items;
    }

    private void calculateTotalPrice() {
        this.totalPrice = items.stream()
                .mapToInt(OrderItem::calculatePrice)
                .sum();
        this.finalPrice = totalPrice;
    }

    public void applyCoupon(Coupon coupon) {
        this.discountPrice = coupon.calculateDiscount(totalPrice);
        this.finalPrice = totalPrice - discountPrice;
    }
}
```
```java
public class OrderItem {

    private Long productId;
    private int price;
    private int quantity;

    public int calculatePrice() {
        return price * quantity;
    }
}
```

Service는 훨씬 단순해집니다.

```java
@Service
@RequiredArgsConstructor
public class OrderService {

    private final ProductRepository productRepository;
    private final CouponRepository couponRepository;
    private final OrderRepository orderRepository;

    @Transactional
    public void order(OrderCommand command) {
        List<Product> products = productRepository.findAllByIds(command.productIds());
        List<OrderItem> orderItems = OrderItem.from(products, command.items());

        Coupon coupon = couponRepository.findById(command.couponId());

        Order order = Order.create(command.memberId(), orderItems, coupon);

        orderRepository.save(order);
    }
}
```

Service는 흐름을 조정하고, 실제 계산과 정책 판단은 도메인 객체가 담당합니다.

### 도메인 오브젝트 사용의 문제점 

다만 도메인 오브젝트 중심 구조가 항상 쉬운 것은 아닙니다.

특히 MyBatis처럼 SQL Mapper 중심으로 개발하는 경우, 도메인 객체를 풍성하게 만들기 어려울 수 있습니다.

예를 들어 MyBatis는 보통 DB 조회 결과를 객체에 매핑합니다.

```java
Order order = orderMapper.selectOrder(orderId);
List<OrderItem> items = orderMapper.selectOrderItems(orderId);
```

이때 Order와 OrderItem의 관계를 자연스럽게 복원하거나, 객체 그래프를 구성하는 작업이 번거로울 수 있습니다.

JPA는 연관관계와 변경 감지를 지원하기 때문에 도메인 중심 설계와 궁합이 좋습니다.

반면 MyBatis는 SQL을 직접 제어하기 좋지만, 도메인 객체의 생명주기나 관계 관리는 개발자가 직접 신경 써야 합니다.

그래서 실무에서는 완전한 도메인 모델보다는 다음과 같은 절충이 많이 사용됩니다.

- 복잡한 쓰기 로직은 도메인 객체에 둡니다.
- 단순 조회는 DTO로 바로 조회합니다.
- 복잡한 통계/목록 조회는 SQL을 적극 활용합니다.

### 빈약한 도메인 오브젝트 방식 

빈약한 도메인 모델은 객체가 상태만 가지고, 행동은 Service에 몰려 있는 구조입니다.

```java
public class Member {
private Long id;
private int point;

    public int getPoint() {
        return point;
    }

    public void setPoint(int point) {
        this.point = point;
    }
}
```

포인트 차감 로직이 Service에 있습니다.

```java
@Service
public class PointService {

    public void usePoint(Member member, int amount) {
        if (member.getPoint() < amount) {
            throw new IllegalArgumentException("포인트가 부족합니다.");
        }

        member.setPoint(member.getPoint() - amount);
    }
}
```

이 방식은 단순하지만, Member의 핵심 규칙이 Member 밖에 존재합니다.

더 나은 방식은 Member가 자신의 포인트를 직접 관리하게 하는 것입니다.

```java
public class Member {

    private Long id;
    private int point;

    public void usePoint(int amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("사용 포인트는 0보다 커야 합니다.");
        }

        if (point < amount) {
            throw new IllegalArgumentException("포인트가 부족합니다.");
        }

        this.point -= amount;
    }
}
```

Service는 이렇게 바뀝니다.

```java
@Service
@RequiredArgsConstructor
public class PointService {

    private final MemberRepository memberRepository;

    @Transactional
    public void usePoint(Long memberId, int amount) {
        Member member = memberRepository.findById(memberId);
        member.usePoint(amount);
        memberRepository.save(member);
    }
}
```

이제 “포인트를 사용할 수 있는 조건”은 Member 내부에 있습니다.

### 풍성한 도메인 오브젝트 방식 

풍성한 도메인 모델은 객체가 자신의 상태와 로직을 함께 가집니다.

예를 들어 쿠폰 발급 도메인을 생각해보면, 쿠폰은 다음 규칙을 가질 수 있습니다.

- 발급 가능 기간 안에만 발급 가능하다.
- 최대 발급 수량을 초과할 수 없다.
- 이미 발급된 쿠폰은 다시 발급할 수 없다.

빈약한 방식에서는 Service가 모든 조건을 검사한다.

```java
public void issueCoupon(Long couponId, Long memberId) {
    Coupon coupon = couponRepository.findById(couponId);

    if (LocalDateTime.now().isBefore(coupon.getIssueStartAt())) {
        throw new IllegalStateException("아직 발급 기간이 아닙니다.");
    }

    if (LocalDateTime.now().isAfter(coupon.getIssueEndAt())) {
        throw new IllegalStateException("발급 기간이 종료되었습니다.");
    }

    if (coupon.getIssuedCount() >= coupon.getMaxIssueCount()) {
        throw new IllegalStateException("쿠폰이 모두 소진되었습니다.");
    }

    boolean alreadyIssued = couponIssueRepository.exists(couponId, memberId);
    if (alreadyIssued) {
        throw new IllegalStateException("이미 발급된 쿠폰입니다.");
    }

    coupon.increaseIssuedCount();
    couponIssueRepository.save(new CouponIssue(couponId, memberId));
}

```

풍성한 방식에서는 쿠폰 발급 가능 여부를 Coupon이 판단합니다.

```java
public class Coupon {

    private Long id;
    private LocalDateTime issueStartAt;
    private LocalDateTime issueEndAt;
    private int maxIssueCount;
    private int issuedCount;

    public void issue(LocalDateTime now) {
        validateIssuePeriod(now);
        validateIssueQuantity();

        this.issuedCount++;
    }

    private void validateIssuePeriod(LocalDateTime now) {
        if (now.isBefore(issueStartAt)) {
            throw new IllegalStateException("아직 발급 기간이 아닙니다.");
        }

        if (now.isAfter(issueEndAt)) {
            throw new IllegalStateException("발급 기간이 종료되었습니다.");
        }
    }

    private void validateIssueQuantity() {
        if (issuedCount >= maxIssueCount) {
            throw new IllegalStateException("쿠폰이 모두 소진되었습니다.");
        }
    }
}
```

Service는 이렇게 정리됩니다.

```java
@Transactional
public void issueCoupon(Long couponId, Long memberId) {
Coupon coupon = couponRepository.findByIdForUpdate(couponId);

    if (couponIssueRepository.exists(couponId, memberId)) {
        throw new IllegalStateException("이미 발급된 쿠폰입니다.");
    }

    coupon.issue(LocalDateTime.now());

    couponIssueRepository.save(new CouponIssue(couponId, memberId));
    couponRepository.save(coupon);
}
```

Service는 여전히 필요한 일을 합니다.

- 쿠폰 조회
- 중복 발급 여부 확인
- 트랜잭션 관리
- 저장 처리

하지만 쿠폰의 핵심 규칙은 Coupon 객체 내부로 들어갑니다.

### 도메인 계층 방식 

도메인 계층 방식은 Service와 Domain의 역할을 더 명확히 나누는 방식입니다.

```
Controller
↓
Application Service
↓
Domain Object / Domain Service
↓
Repository
```

Application Service는 유스케이스를 조정합니다.

```java
@Service
@RequiredArgsConstructor
public class OrderApplicationService {

    private final MemberRepository memberRepository;
    private final ProductRepository productRepository;
    private final CouponRepository couponRepository;
    private final OrderRepository orderRepository;

    @Transactional
    public void placeOrder(PlaceOrderCommand command) {
        Member member = memberRepository.findById(command.memberId());
        List<Product> products = productRepository.findAllByIds(command.productIds());
        Coupon coupon = couponRepository.findById(command.couponId());

        Order order = Order.place(member, products, coupon);

        orderRepository.save(order);
    }
}
```

Domain은 핵심 비즈니스 로직을 담당합니다.

```java
public class Order {

    public static Order place(Member member, List<Product> products, Coupon coupon) {
        List<OrderItem> items = products.stream()
                .map(OrderItem::from)
                .toList();

        Order order = new Order(member.getId(), items);
        order.applyCoupon(coupon);
        order.validateMinimumOrderAmount();

        return order;
    }

    private void validateMinimumOrderAmount() {
        if (finalPrice < 1000) {
            throw new IllegalStateException("최소 주문 금액을 만족하지 않습니다.");
        }
    }
}
```

이렇게 하면 Service가 지나치게 비대해지는 것을 막을 수 있습니다.

### DTO와 리포트 쿼리 

오브젝트 중심 아키텍처를 사용한다고 해서 모든 조회를 도메인 객체로 해야 하는 것은 아닙니다.

특히 목록 조회, 검색, 통계, 관리자 리포트 화면은 DTO와 SQL을 사용하는 편이 더 낫습니다.

예를 들어 주문 상세 화면을 조회한다고 해봅시다.

```java
public OrderDetailResponse getOrderDetail(Long orderId) {
    Order order = orderRepository.findById(orderId);
    Member member = memberRepository.findById(order.getMemberId());
    List<OrderItem> items = orderItemRepository.findAllByOrderId(orderId);

    return OrderDetailResponse.of(order, member, items);
}
```

이 방식은 객체 중심으로 보이지만 조회 목적에서는 불필요하게 여러 번 DB를 조회할 수 있습니다.

조회 전용 DTO를 사용하면 더 단순해집니다.

```java
public OrderDetailResponse getOrderDetail(Long orderId) {
    return orderQueryRepository.findOrderDetail(orderId);
}
```
```sql
SELECT
o.order_id,
o.order_status,
m.name AS member_name,
o.total_price,
o.discount_price,
o.final_price
FROM orders o
JOIN member m ON o.member_id = m.member_id
WHERE o.order_id = #{orderId}
```

즉, 쓰기와 조회를 다르게 설계할 수 있습니다.

쓰기 작업
- 도메인 객체 중심
- 비즈니스 규칙 중요
- 트랜잭션 중요

조회 작업
- DTO 중심
- 화면 요구사항 중요
- 쿼리 성능 중요

이 관점은 실무에서 굉장히 중요합니다.

## 9.3.4 스프링 애플리케이션을 위한 아키텍처 설계 

### 계층형 아키텍처 

스프링은 계층형 아키텍처를 쉽게 적용할 수 있도록 여러 기능을 제공합니다.

```
@Controller / @RestController
@Service
@Repository
@Component
@Transactional
```

예를 들어 Service에 @Transactional을 적용하면 하나의 유스케이스를 트랜잭션 단위로 묶을 수 있습니다.

```java
@Service
@RequiredArgsConstructor
public class OrderService {

    private final OrderRepository orderRepository;
    private final PointRepository pointRepository;

    @Transactional
    public void order(OrderCommand command) {
        pointRepository.usePoint(command.memberId(), command.pointAmount());
        orderRepository.save(command.toOrder());
    }
}
```

이때 Service는 트랜잭션 경계가 되기 좋습니다.
Controller나 Repository보다 Service가 유스케이스 단위를 가장 잘 표현하기 때문입니다.

### 정보 전송 아키텍처 

계층 간 데이터를 어떻게 전달할지도 중요합니다.

Controller에서 받은 요청 객체를 그대로 도메인이나 Repository까지 넘기는 것은 좋지 않습니다.

```java
@PostMapping("/orders")
public ResponseEntity<Void> order(@RequestBody OrderRequest request) {
    orderService.order(request);
    return ResponseEntity.ok().build();
}
```

이렇게 하면 Service가 웹 요청 객체에 의존하게 됩니다.

더 나은 방식은 Request를 Command로 변환하는 것입니다.

```java
public record OrderRequest(
    List<OrderItemRequest> items,
    Long couponId,
    int usePoint
) {
    public OrderCommand toCommand(Long memberId) {
        return new OrderCommand(memberId, items, couponId, usePoint);
    }
}
```
```java
public record OrderCommand(
    Long memberId,
    List<OrderItemRequest> items,
    Long couponId,
    int usePoint
) {
}
```
```java
@PostMapping("/orders")
public ResponseEntity<Void> order(
    @RequestHeader Long memberId,
    @RequestBody OrderRequest request
) {
    orderService.order(request.toCommand(memberId));
    return ResponseEntity.ok().build();
}
```

정리하면 다음과 같습니다.

Request DTO
- 웹 요청을 표현
- Controller 계층에서 사용

Command
- 애플리케이션 유스케이스 입력값
- Service 계층에서 사용

Domain
- 핵심 비즈니스 상태와 로직
- 도메인 계층에서 사용

Response DTO
- 웹 응답을 표현
- Controller 계층에서 반환

### 상태 관리와 빈 스코프 

스프링 빈은 기본적으로 싱글톤입니다.

즉, 하나의 Service 객체가 여러 요청에서 공유됩니다.

그래서 Service에 요청별 상태를 필드로 저장하면 위험합니다.

```java
@Service
public class OrderService {

    private Long currentMemberId;

    public void order(Long memberId) {
        this.currentMemberId = memberId;

        // 주문 처리
    }
}
```

이 코드는 멀티스레드 환경에서 문제가 생길 수 있습니다.
여러 사용자의 요청이 동시에 들어오면 currentMemberId 값이 섞일 수 있기 때문입니다.

따라서 스프링 빈은 기본적으로 상태를 가지지 않도록 설계해야 합니다.

좋은 방식은 요청별 상태를 메서드 파라미터나 지역 변수로 다루는 것입니다.

```java
@Service
public class OrderService {

    public void order(Long memberId) {
        Long currentMemberId = memberId;

        // 주문 처리
    }
}
```

상태가 꼭 필요하다면 Request Scope, Session Scope, ThreadLocal 등을 사용할 수 있지만, 대부분의 Service Bean은 stateless하게 유지하는 것이 좋습니다.

### 서드파티 프레임워크, 라이브러리 적용

스프링은 다양한 외부 기술을 애플리케이션 아키텍처 안에 자연스럽게 통합할 수 있습니다.

예를 들어 MyBatis를 사용하면 Repository 또는 Mapper 계층에서 SQL을 관리할 수 있습니다.

```java
@Repository
@RequiredArgsConstructor
public class OrderRepository {

    private final OrderMapper orderMapper;

    public void save(Order order) {
        orderMapper.insert(order);
    }

    public Order findById(Long orderId) {
        return orderMapper.selectById(orderId);
    }
}
```
```java
@Mapper
public interface OrderMapper {

    void insert(Order order);

    Order selectById(Long orderId);
}
```

Redis, Kafka, 외부 API도 마찬가지입니다.
중요한 것은 외부 기술이 비즈니스 로직을 침범하지 않게 하는 것입니다.

나쁜 예시는 Service 곳곳에 외부 API 호출 코드가 직접 흩어지는 것입니다.

```java
@Service
public class PaymentService {

    public void pay(PaymentCommand command) {
        RestTemplate restTemplate = new RestTemplate();

        // 외부 결제 API 호출
        restTemplate.postForObject("https://payment-api/pay", command, String.class);
    }
}
```

더 나은 방식은 외부 연동을 별도 컴포넌트로 분리하는 것입니다.

```java
@Component
public class PaymentClient {

    public PaymentResult requestPayment(PaymentCommand command) {
        // 외부 결제 API 호출
        return new PaymentResult();
    }
}
```
```java
@Service
@RequiredArgsConstructor
public class PaymentService {

    private final PaymentClient paymentClient;

    public void pay(PaymentCommand command) {
        PaymentResult result = paymentClient.requestPayment(command);

        if (!result.isSuccess()) {
            throw new IllegalStateException("결제 실패");
        }
    }
}
```

이렇게 하면 외부 API 변경이 PaymentClient 안에서만 처리됩니다.

## 정리

1. 애플리케이션은 계층별로 관심사를 분리해야 합니다.
2. Controller는 웹 요청과 응답을 담당합니다.
3. Service는 유스케이스 흐름과 트랜잭션을 담당합니다.
4. Repository는 데이터 접근을 담당합니다.
5. 모든 로직을 SQL이나 Service에 몰아넣으면 유지보수가 어려워집니다.
6. 핵심 비즈니스 규칙은 도메인 객체가 책임지도록 설계하는 것이 좋습니다.
7. 다만 조회성 기능은 DTO와 SQL을 활용하는 것이 현실적입니다.
8. 스프링 빈은 기본적으로 싱글톤이므로 상태를 가지지 않도록 설계해야 합니다.
9. 외부 프레임워크나 라이브러리는 비즈니스 로직과 분리해서 적용해야 합니다.
