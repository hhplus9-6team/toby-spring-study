# 8.2 스프링의 목적

## 서론

스프링은 단순한 기술 프레임워크가 아니라,
엔터프라이즈 애플리케이션 개발에서 발생하는 복잡함을 해결하기 위해 등장했습니다.

이 장에서는

왜 스프링이 필요했는지, 그리고 어떤 방식으로 복잡함을 해결하는지

를 정리합니다.

## 8.2.1 엔터프라이즈 개발의 복잡함

### 1) 복잡함의 근본적인 이유

엔터프라이즈 애플리케이션은 본질적으로 복잡합니다.

- 다양한 비즈니스 요구사항
- 대규모 데이터 처리
- 트랜잭션 관리
- 외부 시스템 연동

즉,

비즈니스 자체가 이미 복잡합니다. 이건 절대 제거할 수 없는 복잡함입니다.

### 2) 복잡함을 가중시키는 원인

문제는 여기에 기술적인 복잡함까지 추가된다는 것입니다.

예:

```java
public void upgradeLevels() {
transactionManager.begin();

    try {
        userDao.update();
        transactionManager.commit();
    } catch (Exception e) {
        transactionManager.rollback();
    }
}
```

문제:
- 비즈니스 로직 + 기술 코드 혼합

예:

```java
Connection c = DriverManager.getConnection(...);
```

문제:

- 특정 기술(JDBC)에 강하게 의존


## 8.2.2 복잡함을 해결하려는 도전

### 1) 제거될 수 없는 근본적인 복잡함

비즈니스 로직의 복잡함은 줄일 수 없습니다.

우리가 할 수 있는 건 기술적 복잡함을 줄이는 것입니다.

### 2) 실패한 해결책: EJB

EJB는 엔터프라이즈 기능을 제공하려 했지만, 오히려 복잡함을 증가시켰습니다.

특징:
- 컨테이너 종속
- 복잡한 인터페이스 구조
- 테스트 어려움


```java
public class UserServiceBean implements SessionBean {
}
```
문제:
- POJO 아님
- 특정 기술에 강하게 결합


### 3) 비침투적인 해결책: 스프링

스프링은 다른 접근을 선택했습니다.

애플리케이션 코드를 기술에 종속시키지 않습니다.

예: POJO

```java
public class UserService {

    public void upgradeLevels() {
        userDao.update();
    }
}
```

특징:
- 특정 프레임워크 의존 없음

>핵심 개념: 비침투성 (Non-invasive)
>
>코드가 프레임워크에 침투되지 않습니다


## 8.2.3 복잡함을 상대하는 스프링의 전략

### 1) 기술적 복잡함을 상대하는 전략

#### (1) 서비스 추상화

```
MailSender → JavaMailSenderImpl
```

효과:
- 기술 변경 자유

#### (2) AOP

```java
@Transactional
public void upgradeLevels() {
    userDao.update();
}
```

효과:
- 트랜잭션 코드 제거


### 2) 비즈니스 로직의 복잡함을 상대하는 전략

핵심은 비즈니스 로직을 깨끗하게 유지하는 것입니다.

예:
```java
public void upgradeLevels() {
    if (user.canUpgrade()) {
        user.upgrade();
    }
}
```

특징:
- 기술 코드 없음
- 로직만 존재

### 3) 핵심 도구: 객체지향과 DI

스프링은 복잡함을 해결하기 위해 특별한 기술이 아니라 객체지향 원칙을 활용합니다.

DI 예시:

```java
public class UserService {

    private MailSender mailSender;

    public void setMailSender(MailSender mailSender) {
        this.mailSender = mailSender;
    }
}
```

효과:

- 결합도 감소
- 테스트 가능
- 확장 가능

## 핵심 정리

스프링은 복잡함을 제거하지 않습니다. 대신 복잡함을 잘 분리하고 관리합니다.

```
비즈니스 복잡함 → 그대로 유지
기술 복잡함 → 분리 / 추상화
```

>스프링의 목적은 엔터프라이즈 개발의 복잡함을 제거하는 것이 아니라, 기술적 복잡함을 분리하고 관리하여 객체지향적으로 해결할 수 있도록 돕는 것입니다.
