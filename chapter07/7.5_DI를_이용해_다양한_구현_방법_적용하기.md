# 7.5 DI를 이용해 다양한 구현 방법 적용하기

## 서론

7.4에서는 UpdatableSqlRegistry를 도입하여 SQL을 조회뿐 아니라 수정할 수 있는 구조로 확장했습니다.

하지만 이 구현은 기능적으로는 동작하더라도, 실제 운영 환경에서 사용하기에는 아직 부족한 점이 있습니다.

7.5에서는 이 SQL 레지스트리를 더 현실적인 방향으로 개선해 나갑니다.

먼저 ConcurrentHashMap을 사용해 동시성 문제를 완화하고, 이후에는 검색 성능, 안정적인 변경 처리, 확장성까지 고려해 내장형 데이터베이스를 사용하는 방식으로 발전시킵니다.

즉, 7.5는 단순한 기능 추가가 아니라 **“동작하는 구조를 실제로 쓸 수 있는 구조로 개선하는 과정”** 입니다.

## 7.5.1 ConcurrentHashMap을 이용한 수정 가능 SQL 레지스트리

## 수정 가능 SQL 레지스트리 테스트

토비는 먼저 UpdatableSqlRegistry가 가져야 할 기능을 테스트로 정의합니다.

이 테스트는 구현체와 무관하게 SQL 등록, 조회, 수정이 정상 동작하는지를 검증하기 위한 역할을 합니다.

```java
public abstract class ConcurrentHashMapUpdatableSqlRegistryTest {

    protected UpdatableSqlRegistry sqlRegistry;

    @Test
    public void registerAndFind() {
        sqlRegistry.registerSql("KEY1", "SQL1");
        sqlRegistry.registerSql("KEY2", "SQL2");

        assertThat(sqlRegistry.findSql("KEY1")).isEqualTo("SQL1");
        assertThat(sqlRegistry.findSql("KEY2")).isEqualTo("SQL2");
    }

    @Test
    public void updateSql() {
        sqlRegistry.registerSql("KEY1", "SQL1");
        sqlRegistry.updateSql("KEY1", "SQL2");

        assertThat(sqlRegistry.findSql("KEY1")).isEqualTo("SQL2");
    }
}
```

이렇게 테스트를 먼저 분리해두면, 이후에 ConcurrentHashMap 기반 구현체든 DB 기반 구현체든 같은 테스트를 재사용하면서 구현체의 품질을 검증할 수 있습니다.

### 수정 가능 SQL 레지스트리 구현

가장 먼저 생각할 수 있는 구현은 메모리 기반 저장소입니다.

다만 단순한 HashMap은 멀티스레드 환경에서 안전하지 않기 때문에, ConcurrentHashMap을 사용해서 구현을 진행합니다.

```java
public class ConcurrentHashMapSqlRegistry implements UpdatableSqlRegistry {

    private final Map<String, String> sqlMap = new ConcurrentHashMap<>();

    @Override
    public void registerSql(String key, String sql) {
        sqlMap.put(key, sql);
    }

    @Override
    public String findSql(String key) {
        String sql = sqlMap.get(key);
        if (sql == null) {
            throw new IllegalArgumentException("No SQL found for key: " + key);
        }
        return sql;
    }

    @Override
    public void updateSql(String key, String sql) {
        if (sqlMap.replace(key, sql) == null) {
            throw new IllegalArgumentException("No SQL found for key: " + key);
        }
    }
}
```

ConcurrentHashMap은 멀티스레드 환경에서 최소한의 동시성 보장을 제공하고, 성능도 크게 나쁘지 않습니다. 

그래서 간단한 수정 가능 SQL 레지스트리 구현체로는 충분히 유용합니다.

하지만 저장되는 데이터가 많아질수록 관리가 어려워지고, 단순 Map 기반이라 인덱스를 이용한 최적화된 검색이 불가능합니다.

잦은 조회와 변경이 동시에 일어나는 환경에서는 더 정교한 저장소가 필요합니다.

즉, ConcurrentHashMap은 “기능 구현”에는 적절하지만, “운영 환경의 안정적인 저장소”로 보기에는 한계가 있습니다.

## 7.5.2 내장형 데이터베이스를 이용한 SQL 레지스트리

### 왜 데이터베이스를 사용하는가

책에서 여기서 강조하는 이유는 단순히 “Map은 메모리라서 날아가니까” 수준이 아닙니다.

더 중요한 건 검색과 변경을 더 안정적이고 효율적으로 처리하기 위해서입니다.

데이터베이스를 사용하면 다음과 같은 장점이 있습니다.

- 인덱스를 이용한 최적화된 검색
- 동시에 많은 요청이 들어와도 안정적으로 조회/변경 처리 가능
- 데이터 양이 늘어나도 관리 가능
- 저장소 자체가 데이터 관리용으로 설계되어 있음

즉, SQL을 단순한 설정값이 아니라 조회와 수정이 반복되는 관리 대상 데이터로 바라보게 되는 것입니다.

### 스프링의 내장형 DB 지원

```java
EmbeddedDatabase db = new EmbeddedDatabaseBuilder()
    .setType(EmbeddedDatabaseType.H2)
    .addScript("classpath:schema.sql")
    .build();
```

스프링은 테스트나 경량 저장소 용도로 사용할 수 있는 내장형 DB를 쉽게 만들 수 있도록 지원합니다.

이 기능을 사용하면 별도의 외부 DB를 준비하지 않아도 애플리케이션 내부에서 손쉽게 데이터베이스 기반 SQL 레지스트리를 구성할 수 있습니다.

### 내장형 DB를 이용한 SqlRegistry 만들기

```java
public class EmbeddedDbSqlRegistry implements UpdatableSqlRegistry {

    private JdbcTemplate jdbc;

    public void setDataSource(DataSource dataSource) {
        this.jdbc = new JdbcTemplate(dataSource);
    }

    @Override
    public void registerSql(String key, String sql) {
        jdbc.update("insert into sqlmap(key_, sql_) values(?, ?)", key, sql);
    }

    @Override
    public String findSql(String key) {
        return jdbc.queryForObject(
            "select sql_ from sqlmap where key_ = ?",
            String.class,
            key
        );
    }

    @Override
    public void updateSql(String key, String sql) {
        jdbc.update("update sqlmap set sql_ = ? where key_ = ?", sql, key);
    }
}
```

DB 기반 구현은 JdbcTemplate을 사용해 SQL을 등록/조회/수정하는 형태로 만들 수 있습니다.

### UpdatableSqlRegistry 테스트 코드의 재사용

이 구조의 중요한 장점은, 앞에서 만든 공통 테스트를 그대로 재사용할 수 있다는 점입니다.

즉,
- ConcurrentHashMapSqlRegistry
- EmbeddedDbSqlRegistry

서로 구현은 다르지만 둘 다 같은 UpdatableSqlRegistry 계약을 따르기 때문에 동일한 테스트를 적용할 수 있습니다.

이건 단순한 테스트 재사용을 넘어 인터페이스 기반 설계가 얼마나 강력한지 보여주는 예시입니다.

### XML 설정을 통한 내장형 DB의 생성과 적용

```xml
<jdbc:embedded-database id="embeddedDatabase" type="H2">
    <jdbc:script location="classpath:schema.sql"/>
</jdbc:embedded-database>

<bean id="sqlRegistry" class="com.example.EmbeddedDbSqlRegistry">
    <property name="dataSource" ref="embeddedDatabase"/>
</bean>
```

스프링 XML 설정을 이용하면 내장형 DB와 레지스트리를 깔끔하게 연결할 수 있습니다.

이렇게 하면 SQL 레지스트리의 저장소 구현을 코드 수정 없이 설정만으로 바꿀 수 있습니다.

## 7.5.3 트랜잭션 적용

### 다중 SQL 수정에 대한 트랜잭션 테스트

SQL 하나를 수정하는 것은 단순합니다. 하지만 여러 SQL을 한꺼번에 수정해야 하는 경우, 중간에 하나라도 실패하면 문제가 생깁니다.

예를 들어 다음과 같은 상황을 생각할 수 있습니다.

```java
sqlRegistry.updateSql("KEY1", "SQL1");
sqlRegistry.updateSql("KEY2", "INVALID SQL");
sqlRegistry.updateSql("KEY3", "SQL3");
```

이 경우 중간에 실패하면 일부 SQL만 반영될 수 있습니다.

따라서 전체 성공 또는 전체 실패가 보장되어야 합니다.

이 요구를 검증하기 위해 트랜잭션 테스트를 작성합니다.

### 코드를 이용한 트랜잭션 적용

```java
public void updateSql(Map<String, String> sqlmap) {
TransactionStatus status = transactionManager.getTransaction(new DefaultTransactionDefinition());

    try {
        for (Map.Entry<String, String> entry : sqlmap.entrySet()) {
            sqlRegistry.updateSql(entry.getKey(), entry.getValue());
        }
        transactionManager.commit(status);
    } catch (RuntimeException e) {
        transactionManager.rollback(status);
        throw e;
    }
}
```

여러 SQL 변경 작업을 하나의 트랜잭션으로 묶으면, 중간에 실패하더라도 전체가 롤백됩니다.

## 정리

7.5는 7.4에서 만든 “수정 가능한 SQL 레지스트리”를 실제 환경에서도 사용할 수 있도록 한 단계 발전시키는 과정을 거쳤습니다.

ConcurrentHashMap으로 동시성을 보완하고 내장형 데이터베이스로 검색 성능과 안정성을 높이며 트랜잭션을 적용해 다중 수정 작업의 일관성까지 보장합니다.

즉, 7.5의 핵심은 단순히 SQL을 수정 가능하게 만드는 것이 아니라 SQL을 안전하고 확장 가능하게 관리할 수 있는 저장소 구조로 발전시키는 것입니다.
