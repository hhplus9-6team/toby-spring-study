# 7.6 스프링 3.1의 DI

> 스프링이 처음 등장한 이후 DI 활용 방식에도 큰 변화가 있었다. 대표적으로 **애노테이션**을 적극 활용하여 DI 관련 정보를 자바 코드에 직접 담게 된 것이다.

## 자바 언어의 변화와 스프링

### 애노테이션의 메타정보 활용

자바 5 이후, 자바 코드가 **실행되는 것이 목적이 아니라, 다른 자바 코드에 의해 데이터처럼 취급**되기도 하는 프로그래밍 방식이 확산되었다. 애노테이션은 이 흐름의 핵심이다.

- 애노테이션은 코드 실행에 직접 참여하지 못한다. 리플렉션 API를 이용해 애노테이션의 메타정보를 조회하고, 이를 참고하여 프레임워크 등이 동작 방식을 결정한다.
- 대신 클래스에 애노테이션 하나를 붙이는 것만으로도 패키지명, 클래스명, 접근 제한자, 상속 관계 등 **풍부한 메타정보를 얻을 수 있다**. XML에서는 이 모든 정보를 일일이 텍스트로 작성해야 한다.

**장점**: 핵심 로직을 담은 자바 코드, IoC 프레임워크, 메타정보가 긴밀하게 결합되어 코드가 간결해진다.  
**단점**: 변경 시 매번 클래스를 새로 컴파일해야 한다. (XML은 텍스트 수정만 하면 됨)

### 정책과 관례를 이용한 프로그래밍

명시적으로 동작 내용을 기술하는 대신, **미리 약속한 규칙(관례)에 따라 프로그램이 동작**하도록 하는 프로그래밍 스타일을 적극 포용하게 됐다.

- 예: XML의 `<bean>` 태그를 만나면 오브젝트를 자동 생성한다는 관례
- 예: `@Transactional`이 메소드와 클래스에 동시에 붙어 있으면 메소드 선언이 우선한다는 정책

코드가 간결해지지만, 관례를 모르면 동작 방식을 파악하기 어렵다는 단점도 있다.


## 7.6.1 자바 코드를 이용한 빈 설정

스프링 3.1에서는 XML로 작성하던 스프링 설정 정보를 자바 코드로 대체할 수 있게 되었다. 기존 XML 설정을 단계적으로 자바 코드로 전환한다.

### 테스트 컨텍스트의 변경

`@ContextConfiguration`에서 XML 위치 대신 자바 설정 클래스를 지정한다. 처음에는 `@ImportResource`로 기존 XML을 가져오면서 하나씩 전환한다.

```java
// 변경 전
@ContextConfiguration(locations = "/test-applicationContext.xml")

// 변경 후
@ContextConfiguration(classes = TestApplicationContext.class)
```

```java
@Configuration
@ImportResource("/test-applicationContext.xml")
public class TestApplicationContext {
}
```

### `<context:annotation-config />` 제거

`@Configuration` 클래스를 사용하면 `@Autowired`, `@PostConstruct` 등을 처리하는 빈 후처리기가 **자동 등록**되므로, XML에서 `<context:annotation-config />`를 제거해도 된다.

### `<bean>`의 전환 - @Bean 메소드

XML의 `<bean>` 정의를 `@Bean` 메소드로 1:1 전환한다.

```xml
<!-- XML -->
<bean id="dataSource" class="org.springframework.jdbc.datasource.SimpleDriverDataSource">
    <property name="driverClass" value="com.mysql.jdbc.Driver" />
    <property name="url" value="jdbc:mysql://localhost/springbook?characterEncoding=UTF-8" />
    <property name="username" value="spring" />
    <property name="password" value="book" />
</bean>
```

```java
// 자바 코드
@Bean
public DataSource dataSource() {
    SimpleDriverDataSource dataSource = new SimpleDriverDataSource();
    dataSource.setDriverClass(com.mysql.jdbc.Driver.class);
    dataSource.setUrl("jdbc:mysql://localhost/springbook?characterEncoding=UTF-8");
    dataSource.setUsername("spring");
    dataSource.setPassword("book");
    return dataSource;
}
```

핵심 규칙:
- **메소드 이름 = 빈 ID** (XML `<bean>`의 id에 해당)
- **반환 타입은 인터페이스**, 지역변수는 구체 클래스로 선언한다. `DataSource` 인터페이스에는 `setUrl()` 같은 수정자가 없으므로, 프로퍼티 설정을 위해 구체 클래스 타입의 지역변수가 필요하다.
- 다른 빈을 참조할 때는 해당 `@Bean` 메소드를 직접 호출한다.

```java
@Bean
public PlatformTransactionManager transactionManager() {
    DataSourceTransactionManager tm = new DataSourceTransactionManager();
    tm.setDataSource(dataSource());  // @Bean 메소드 호출로 DI
    return tm;
}
```

### 전용 태그 전환

#### `<tx:annotation-driven />` -> `@EnableTransactionManagement`

`<tx:annotation-driven />`은 내부적으로 4개의 빈(자동 프록시 생성기, 트랜잭션 속성 소스, 트랜잭션 인터셉터, 어드바이저)을 등록하는데, `@EnableTransactionManagement` 하나로 모두 대체한다.

#### `<jdbc:embedded-database>` -> `EmbeddedDatabaseBuilder`

```java
@Bean
public DataSource embeddedDatabase() {
    return new EmbeddedDatabaseBuilder()
        .setName("embeddedDatabase")
        .setType(HSQL)
        .addScript("classpath:springbook/user/sqlmap/sqlRegistrySchema.sql")
        .build();
}
```

모든 전환이 끝나면 `@ImportResource`를 제거하고, XML 파일도 삭제할 수 있다.

```java
@Configuration
@EnableTransactionManagement
public class TestApplicationContext {
    // 모든 빈이 @Bean 메소드로 정의됨
}
```


## 7.6.2 빈 스캐닝과 자동와이어링

7.6.1에서 XML을 `@Bean` 메소드로 전환했지만, 여전히 모든 빈을 일일이 등록해야 한다. `@Autowired`와 `@Component`를 사용하면 이 과정을 더 간소화할 수 있다.

### @Autowired를 이용한 자동와이어링

`@Autowired`는 스프링이 **타입을 기준**으로 조건에 맞는 빈을 자동으로 찾아서 주입해주는 기능이다. setter 메소드, 필드, 생성자, 일반 메소드에 적용할 수 있다.

- **타입 기준 검색**: 주입 가능한 타입의 빈이 하나이면 해당 빈을 주입
- **동일 타입 빈이 2개 이상**: 프로퍼티(또는 필드) 이름과 동일한 이름을 가진 빈을 찾아서 주입
- **private 필드 주입**: 리플렉션을 이용하므로 setter 없이 private 필드에도 직접 주입 가능

#### setter에 @Autowired

```java
public class UserDaoJdbc implements UserDao {
    private JdbcTemplate jdbcTemplate;

    @Autowired
    public void setDataSource(DataSource dataSource) {
        this.jdbcTemplate = new JdbcTemplate(dataSource);
    }
}
```

`setDataSource()`는 받은 DataSource를 그대로 저장하는 게 아니라 **JdbcTemplate을 생성해서 저장**한다. 이런 경우 필드 주입으로 대체할 수 없다. setter 안의 로직이 실행되어야 하기 때문이다.

#### 필드에 @Autowired

```java
@Autowired
private SqlService sqlService;
```

스프링이 **리플렉션으로 private 필드에 직접 값을 주입**한다. setter 없이도 빈을 주입받을 수 있어 코드가 간결해진다.

### @Resource

`@Autowired`가 **타입 기준**이라면, `@Resource`는 **이름 기준**으로 빈을 찾는다. 같은 타입의 빈이 여러 개일 때 유용하다.

```java
@Resource(name = "transactionManager")
private PlatformTransactionManager transactionManager;
```

### @Component를 이용한 자동 빈 등록

`@Component`가 붙은 클래스는 빈 스캐너에 의해 **자동으로 빈으로 등록**된다. `@Bean` 메소드를 일일이 작성하지 않아도 된다.

```java
@Component
public class UserDaoJdbc implements UserDao {
    @Autowired
    private SqlService sqlService;

    @Autowired
    public void setDataSource(DataSource dataSource) {
        this.jdbcTemplate = new JdbcTemplate(dataSource);
    }
}
```

`@ComponentScan`으로 스캔 대상 패키지를 지정해야 한다. 지정된 패키지와 그 하위 패키지 전체를 스캔한다.

```java
@Configuration
@EnableTransactionManagement
@ComponentScan(basePackages = "springbook.user")
public class AppContext { }
```

**빈 이름 규칙:**
- 기본값: 클래스명의 **첫 글자를 소문자**로 변경 (`UserDaoJdbc` -> `userDaoJdbc`)
- 직접 지정: `@Component("userDao")`

### 스테레오타입 애노테이션

스프링이 제공하는 스테레오타입 애노테이션은 모두 `@Component`를 메타 애노테이션으로 포함하여 자동 빈 등록 대상이 된다.

| 애노테이션 | 용도 |
|---|---|
| `@Repository` | DAO 빈 (데이터 접근 계층) |
| `@Service` | 비즈니스 로직, 트랜잭션 경계 |
| `@Controller` | 웹 컨트롤러 (DispatcherServlet이 인식) |

```java
@Repository
public class UserDaoJdbc implements UserDao { ... }

@Service("userService")
public class UserServiceImpl implements UserService { ... }
```

`UserServiceImpl`처럼 인터페이스 이름과 클래스 이름이 다른 경우, 별도로 이름을 지정하지 않으면 디폴트 빈 이름은 `userServiceImpl`이 된다. 의도한 이름으로 쓰려면 `@Service("userService")`처럼 직접 지정해야 한다.

### 메타 애노테이션

**애노테이션의 정의에 붙는 애노테이션**을 메타 애노테이션이라 한다. 커스텀 스테레오타입 애노테이션을 만들 수도 있다.

```java
@Component
public @interface SnsConnector { }

@SnsConnector
public class TwitterConnector { }  // 자동 빈 등록 대상
```

## 7.6.3 컨텍스트 분리와 @Import

### 테스트용 컨텍스트 분리

운영 환경과 테스트 환경의 빈 설정을 분리한다. 테스트에서만 필요한 빈(예: 테스트용 `MailSender`)은 별도의 설정 클래스로 분리한다.

```java
// 운영용 설정
@Configuration
@EnableTransactionManagement
@ComponentScan(basePackages = "springbook.user")
public class AppContext {
    @Bean
    public DataSource dataSource() { ... }

    @Bean
    public PlatformTransactionManager transactionManager() { ... }
}

// 테스트용 설정
@Configuration
public class TestAppContext {
    @Bean
    public UserService testUserService() {
        return new UserServiceTest.TestUserService();
    }

    @Bean
    public MailSender mailSender() {
        return new DummyMailSender();
    }
}
```

테스트에서는 두 설정 클래스를 모두 지정한다.

```java
@RunWith(SpringJUnit4ClassRunner.class)
@ContextConfiguration(classes = {TestAppContext.class, AppContext.class})
public class UserDaoTest {
    @Autowired UserDao dao;
}
```

### @Import

SQL 서비스처럼 **독립적인 모듈**은 별도의 설정 클래스로 분리하고 `@Import`로 가져온다.

```java
@Configuration
public class SqlServiceContext {
    @Bean
    public SqlService sqlService() {
        OxmSqlService sqlService = new OxmSqlService();
        sqlService.setUnmarshaller(unmarshaller());
        sqlService.setSqlRegistry(sqlRegistry());
        return sqlService;
    }

    @Bean
    public SqlRegistry sqlRegistry() {
        EmbeddedDbSqlRegistry sqlRegistry = new EmbeddedDbSqlRegistry();
        sqlRegistry.setDataSource(embeddedDatabase());
        return sqlRegistry;
    }

    // ...
}
```

```java
@Configuration
@EnableTransactionManagement
@ComponentScan(basePackages = "springbook.user")
@Import(SqlServiceContext.class)
public class AppContext {
    // SQL 서비스 관련 빈은 SqlServiceContext에서 관리
}
```

XML 설정을 가져올 때는 `@ImportResource`, 자바 클래스 설정을 가져올 때는 `@Import`를 사용한다.

## 7.6.4 프로파일

### @Profile

같은 타입이면서 아이디도 같은 빈이 두 개의 설정 클래스에 나뉘어 존재할 때, 실행 환경(테스트, 운영 등)에 따라 빈 구성을 다르게 설정할 수 있다. `@Profile` 애노테이션을 사용하면 된다.

```java
@Configuration
@Profile("test")
public class TestAppContext {
    @Bean
    public UserService testUserService() {
        return new UserServiceTest.TestUserService();
    }

    @Bean
    public MailSender mailSender() {
        return new DummyMailSender();
    }
}

@Configuration
@Profile("production")
public class ProductionAppContext {
    @Bean
    public MailSender mailSender() {
        // 실제 메일 발송 구현
    }
}
```

프로파일이 지정된 설정 클래스는 해당 프로파일이 활성화되지 않으면 **무시**된다. 프로파일이 지정되지 않은 설정 클래스는 항상 적용된다.

### @ActiveProfiles

테스트에서 활성화할 프로파일을 지정한다.

```java
@RunWith(SpringJUnit4ClassRunner.class)
@ActiveProfiles("test")
@ContextConfiguration(classes = AppContext.class)
public class UserServiceTest {
    // ...
}
```

### 컨테이너의 빈 등록 정보 확인

`DefaultListableBeanFactory`를 주입받아 등록된 빈 정보를 확인할 수 있다.

```java
@Autowired
DefaultListableBeanFactory bf;

@Test
public void beans() {
    for (String n : bf.getBeanDefinitionNames()) {
        System.out.println(n + " \t " + bf.getBean(n).getClass().getName());
    }
}
```

### 중첩 클래스를 이용한 프로파일 적용

여러 프로파일 설정 클래스를 **static 중첩 클래스**로 하나의 설정 클래스 안에 통합할 수 있다. static 중첩 클래스로 넣으면 `@Import`에서 해당 클래스를 별도로 지정하지 않아도 자동으로 인식된다.

```java
@Configuration
@EnableTransactionManagement
@ComponentScan(basePackages = "springbook.user")
@Import(SqlServiceContext.class)
public class AppContext {

    // ... 공통 빈 설정 ...

    @Configuration
    @Profile("production")
    public static class ProductionAppContext {
        @Bean
        public MailSender mailSender() {
            // 실제 메일 발송 구현
        }
    }

    @Configuration
    @Profile("test")
    public static class TestAppContext {
        @Bean
        public UserService testUserService() {
            return new UserServiceTest.TestUserService();
        }

        @Bean
        public MailSender mailSender() {
            return new DummyMailSender();
        }
    }
}
```

## 7.6.5 프로퍼티 소스

### @PropertySource

DB 연결 정보같이 환경에 따라 달라지는 값은 프로퍼티 파일로 분리하는 것이 좋다.

`database.properties`:
```properties
db.driverClass=com.mysql.jdbc.Driver
db.url=jdbc:mysql://localhost/springbook?characterEncoding=UTF-8
db.username=spring
db.password=book
```

`@PropertySource`로 프로퍼티 파일을 등록하고, `Environment` 오브젝트로 값을 가져올 수 있다.

```java
@Configuration
@EnableTransactionManagement
@ComponentScan(basePackages = "springbook.user")
@Import(SqlServiceContext.class)
@PropertySource("/database.properties")
public class AppContext {

    @Autowired
    Environment env;

    @Bean
    public DataSource dataSource() {
        SimpleDriverDataSource ds = new SimpleDriverDataSource();
        try {
            ds.setDriverClass(
                (Class<? extends java.sql.Driver>) Class.forName(
                    env.getProperty("db.driverClass")));
        } catch (ClassNotFoundException e) {
            throw new RuntimeException(e);
        }
        ds.setUrl(env.getProperty("db.url"));
        ds.setUsername(env.getProperty("db.username"));
        ds.setPassword(env.getProperty("db.password"));
        return ds;
    }
}
```

`Environment` 오브젝트의 `getProperty()` 메소드는 프로퍼티 이름을 키로 사용해서 값을 가져온다. `@PropertySource`로 등록한 프로퍼티 파일의 값이 `Environment` 오브젝트에 자동으로 추가된다.

### PropertySourcesPlaceholderConfigurer

`@Value` 애노테이션과 `${...}` 치환자를 사용하면 `Environment` 오브젝트를 직접 사용하는 것보다 코드를 더 간결하게 작성할 수 있다.

```java
@Configuration
@PropertySource("/database.properties")
public class AppContext {

    @Value("${db.driverClass}")
    Class<? extends Driver> driverClass;

    @Value("${db.url}")
    String url;

    @Value("${db.username}")
    String username;

    @Value("${db.password}")
    String password;

    @Bean
    public static PropertySourcesPlaceholderConfigurer placeholderConfigurer() {
        return new PropertySourcesPlaceholderConfigurer();
    }

    @Bean
    public DataSource dataSource() {
        SimpleDriverDataSource ds = new SimpleDriverDataSource();
        ds.setDriverClass(this.driverClass);
        ds.setUrl(this.url);
        ds.setUsername(this.username);
        ds.setPassword(this.password);
        return ds;
    }
}
```

주의할 점:
- `PropertySourcesPlaceholderConfigurer`는 `BeanFactoryPostProcessor`이므로 반드시 **static 메소드**로 등록해야 한다. static이 아니면 `@Configuration` 클래스가 너무 일찍 초기화되어 `@Autowired`, `@Value` 등의 애노테이션이 제대로 처리되지 않는다.
- `@Value`는 **타입 변환을 자동으로** 해준다. 예를 들어 `db.driverClass`의 문자열 값을 `Class<? extends Driver>` 타입으로 자동 변환한다.
- `PropertySourcesPlaceholderConfigurer`는 스프링 3.1에서 기존의 `PropertyPlaceholderConfigurer`를 대체한다.


## 7.6.6 빈 설정의 재사용과 @Enable*

### 빈 설정자

SQL 서비스처럼 독립적인 모듈을 라이브러리로 만들어 다른 프로젝트에서도 재사용할 수 있도록 한다. `SqlServiceContext`를 별도 모듈로 분리하면 `@Import`로 간단히 가져다 쓸 수 있다.

하지만 SQL 매핑 파일의 위치처럼 프로젝트마다 달라지는 설정값이 있을 수 있다. 이 경우 `SqlMapConfig` 같은 인터페이스를 정의하고, 설정 클래스에서 이를 구현하게 하여 빈 설정 정보를 외부에서 주입받을 수 있다.

```java
public interface SqlMapConfig {
    Resource getSqlMapResource();
}
```

```java
@Configuration
public class SqlServiceContext {
    @Autowired
    SqlMapConfig sqlMapConfig;

    @Bean
    public SqlService sqlService() {
        OxmSqlService sqlService = new OxmSqlService();
        sqlService.setSqlmap(sqlMapConfig.getSqlMapResource());
        // ...
        return sqlService;
    }
}
```

```java
@Configuration
public class AppContext implements SqlMapConfig {

    @Override
    public Resource getSqlMapResource() {
        return new ClassPathResource("sqlmap.xml", UserDao.class);
    }
}
```

### @Enable* 애노테이션

`@Import`를 직접 쓰는 것보다 의미가 명확한 **전용 애노테이션**을 정의하여 사용할 수 있다.

```java
@Import(value = SqlServiceContext.class)
public @interface EnableSqlService {
}
```

사용:

```java
@Configuration
@EnableTransactionManagement
@ComponentScan(basePackages = "springbook.user")
@EnableSqlService
@PropertySource("/database.properties")
public class AppContext {
    // ...
}
```

`@EnableSqlService`는 `@Import(SqlServiceContext.class)`와 동일하게 동작하지만, **이름 자체로 의미가 명확하게 드러난다**.

엘리먼트를 이용하면 옵션을 지정할 수도 있다.

```java
@Import(value = SqlServiceContext.class)
public @interface EnableSqlService {
    String value() default "";
}
```

스프링이 제공하는 대표적인 `@Enable*` 애노테이션:
- `@EnableTransactionManagement` - `<tx:annotation-driven />` 대체
- `@EnableWebMvc` - `<mvc:annotation-driven />` 대체


## 정리

스프링 3.1은 XML 중심의 설정 방식에서 벗어나 **자바 코드와 애노테이션 기반의 설정 방식**을 본격적으로 지원한다. 7장 전체를 통해 발전시켜온 UserDao, UserService, SqlService의 빈 설정을 단계적으로 자바 코드로 전환하면서 다음과 같은 핵심 기능을 살펴봤다.

| 단계 | 핵심 내용 |
|---|---|
| XML -> Java | `@Configuration` + `@Bean`으로 XML 설정을 자바 코드로 전환 |
| 자동화 | `@Autowired`, `@Component`, `@ComponentScan`으로 자동 와이어링 및 빈 스캐닝 |
| 설정 분리 | `@Import`로 모듈별 설정 클래스 분리, 테스트/운영 설정 분리 |
| 환경별 설정 | `@Profile` + `@ActiveProfiles`로 환경(test/production)별 빈 구성 |
| 외부 설정 | `@PropertySource` + `@Value`로 프로퍼티 파일 외부화 |
| 재사용 | `@Enable*` 커스텀 애노테이션으로 빈 설정 모듈화 및 재사용 |
