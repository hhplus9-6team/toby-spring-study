# 7.1 SQL과 DAO의 분리

## 1. 문제인식

기존 DAO는 SQL을 코드 내부에 직접 포함하고 있었습니다.

```java
public void add(User user) throws SQLException {
    Connection c = dataSource.getConnection();

    PreparedStatement ps =
            c.prepareStatement("insert into users(id, name, password) values(?,?,?)");

    ps.setString(1, user.getId());
    ps.setString(2, user.getName());
    ps.setString(3, user.getPassword());

    ps.executeUpdate();
}
```

이 방식은 다음과 같은 문제를 가집니다.

- SQL 변경 시 DAO 코드 수정 필요
- SQL과 데이터 접근 로직이 섞여 있음 (SRP 위반)
- 확장과 변경에 취약 (OCP 위반)

따라서 SQL을 DAO에서 분리할 필요가 있습니다.

## 2. XML 설정을 이용한 분리

### 1) 개별 SQL 프로퍼티 방식

```xml
<bean id="sqlService" class="java.util.Properties">
    <property name="properties">
        <props>
            <prop key="userAdd">
                insert into users(id, name, password) values(?,?,?)
            </prop>
            <prop key="userGet">
                select * from users where id = ?
            </prop>
        </props>
    </property>
</bean>
```

```java
public class UserDao {

    private Properties sqlService;

    public void setSqlService(Properties sqlService) {
        this.sqlService = sqlService;
    }

    public void add(User user) throws SQLException {
        String sql = sqlService.getProperty("userAdd");

        PreparedStatement ps = connection.prepareStatement(sql);
        // ...
    }
}
```

SQL을 하나씩 프로퍼티로 정의하는 방식입니다.

- 간단하게 SQL을 외부로 분리 가능
- 하지만 SQL이 많아질수록 관리가 어려움

### 2) SQL 맵 프로퍼티 방식

```xml
<util:map id="sqlMap">
    <entry key="userAdd" value="insert into users(id, name, password) values(?,?,?)"/>
    <entry key="userGet" value="select * from users where id = ?"/>
</util:map>
```

```java
public class UserDao {

    private Map<String, String> sqlMap;

    public void setSqlMap(Map<String, String> sqlMap) {
        this.sqlMap = sqlMap;
    }

    public void add(User user) throws SQLException {
        String sql = sqlMap.get("userAdd");
        // ...
    }
}
```

SQL을 하나의 Map 형태로 관리하는 방식입니다.

- SQL을 키 기반으로 관리
- 구조적으로 더 명확함
- 하지만 여전히 타입 안정성이 없음

## 3. SQL 제공 서비스

단순히 Map이나 Properties를 사용하는 것보다 **SQL을 제공하는 역할을 별도의 서비스로 분리**하는 것이 더 좋은 설계입니다.

### 1) SQL 서비스 인터페이스

```java
public interface SqlService {
    String getSql(String key);
}
```

```java
public class UserDao {

    private SqlService sqlService;

    public void setSqlService(SqlService sqlService) {
        this.sqlService = sqlService;
    }

    public void add(User user) throws SQLException {
        String sql = sqlService.getSql("userAdd");
        // ...
    }
}
```

SQL 제공 역할을 인터페이스로 추상화하고, DAO는 이 인터페이스에만 의존합니다.

이제 DAO는 더 이상 SQL의 저장 방식에 대해 알 필요가 없습니다.

### 2) 스프링 설정을 사용하는 단순 SQL 서비스

```java
public class SimpleSqlService implements SqlService {

    private Map<String, String> sqlMap;

    public void setSqlMap(Map<String, String> sqlMap) {
        this.sqlMap = sqlMap;
    }

    @Override
    public String getSql(String key) {
        String sql = sqlMap.get(key);
        if (sql == null) {
            throw new IllegalArgumentException("No SQL found for key: " + key);
        }
        return sql;
    }
}
```

```xml
<util:map id="sqlMap">
    <entry key="userAdd" value="insert into users(id, name, password) values(?,?,?)"/>
    <entry key="userGet" value="select * from users where id = ?"/>
</util:map>

<bean id="sqlService" class="com.example.SimpleSqlService">
    <property name="sqlMap" ref="sqlMap"/>
</bean>
```

가장 간단한 구현은 Map을 내부에 사용하는 방식입니다.

## 4. 개선된 구조

```
UserDao
  ↓
SqlService (인터페이스)
  ↓
SimpleSqlService (구현체)
  ↓
Map / XML
```

## 5. 개선 효과

### 1) SRP 만족
- DAO: 데이터 접근 로직
- SqlService: SQL 제공

### 2) OCP 만족
- SQL 저장 방식 변경 시 DAO 수정 없음

### 3) 확장성 확보
- XML
- DB
- 캐싱
- 파일 기반

## 6. 정리

SQL을 DAO에서 분리하고, 이를 제공하는 서비스를 도입하면 DAO는 데이터 접근 로직에만 집중할 수 있고 SQL은 독립적으로 관리될 수 있습니다.

이는 변경에 유연하고 확장 가능한 구조를 만듭니다.
