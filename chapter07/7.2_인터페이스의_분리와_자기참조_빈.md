## 1. JAXB(Java Architecture for XML Binding) 적용
* XML 파일을 자바 객체로 바인딩(=언마샬링) 하여 사용할 수 있도록 해주는 기술
* 사용 방법에 대한 기술은 생략하고 적용한 것만 가져옴
```java
public class XmlSqlService implements SqlService {
    private Map<String, String> sqlMap = new HashMap<String>();
    
    public XmlSqlService() {
        String contextPath = SqlmapClass.getPackage().getName();
        try {
            JAXBContext context = JAXBContext.newInstance(contextPath);
            Unmarshaller unmarshaller = context.createUnmarshaller();
            InputStream is = UserDao.class.getResourceAsStream("sqlmap.xml"); // UserDao와 같은 클래스패스의 sqlmap.xml 파일 반환
            Sqlmap sqlmap = unmarshaller.unmarshal(is);

            for(SqlType sql : sqlmap.getSql()){
                sqlMap.put(sql.getKey(), sql.getValue());
            }
        } catch(Exxception e){
            throw new RuntimeException(e);
        }
    }
    
    public String getSql(String key) {
        // 생략
    }
} 
```
### 장점
* JAXB 를 사용하기 전과 달리 트리 구조의 객체를 사용하 듯 XML 내용을 꺼내올 수 있게 되어 편리해진다.
### 문제점
1. Bean이 생성될 때 SQL에 대한 값을 읽어와 상태값으로 세팅해주어야하기에 생성자 내부에 넣어줬지만 이는 안티패턴.
  * 생성자에서 발생하는 예외는 다루기 어렵기에 예외가 발생할 수 있는 복잡한 초기화 작업은 지양하는게 좋고
  * 상속할 때 불편하고
  * 보안에 문제가 생길 수 있기 때문이다
2. 해당 클래스가 수정되는 이유가 한가지가 아님(단일 책임 원칙 위반)
   * SQL 자체가 변하지 않았는데도, XML 포맷 외의 다른 파일을 사용하면 새로운 SqlService 구현체를 만들어야 함.
   * 가져온 정보가 Map이 아닌 다른 컬랙션으로 수정되야하는 등 XML 자체의 기술 변화.

## 2. @PostConstructor 적용
### 2.1. @PostConstructor 란
* 스프링이 제공하는 애너테이션을 이용한 빈 후처리기 일종으로, JDK 6에 포함된다.
* DI 작업을 마친 뒤에 `@PostContruct`가 붙은 메서드를 자동으로 실행시킨다
* 필수 설정:
```xml
    <tx:annotation-drive/> <!--@Transactional이 붙은 타입과 메소드에 부가기능을ㅊ 담은 프록시를 추가하도록 만들어주는 후처리기를 등록-->
    <context:annotation-config/> <!--코드의 애너테이션을 이용해서 부가적인 빈 설정 또는 초기화 작업을 해주는 후처리기 등록-->
```

## 3. 책임 분리
### 3.1. XmlSqlService 의 역할에 따라 인터페이스 설계
분리 가능한 관심사를 정의해서 **전략패턴**을 적용하여 별도의 오브젝트로 분리. 
1. 외부의 리소스(XML파일)로부터 SQL 정보 읽어오기  -> `SqlReader`
2. 읽어온 SQL을 보관해두고 있다 필요할때 제공해주기 -> `SqlRegistry`

### 3.2. SqlReader -> SqlRegistry 사이의 데이터 전달은 어떻게 할까?
데이터를 객체로 전달할 수도 있지만 전략패턴의 부모클래스에서 사용하는 것이 아니라면 불필요한 작업이다.
이는 `SqlReader`에 `SqlRegistry`를 전달함으로써 해결할 수 있다.
* 이게 아니라
   ```java
   Map<String, String> sqls = sqlReader.readSql();
   sqlRegistry.addSqls(sqls);
   ```
* 이렇게
   ```java
   sqlReader.readSql(sqlRegistry);
   ```
   ```java
  public interface SqlRegistry {
    void registerSql(String key, String sql);
  }   
   ```

### 3.3. XmlSqlService 에 설계 적용해보기 (feat.자기참조빈)
* 설계는 했으나 `XmlSqlService`를 별개의 객체로 분리하기 어려운 상황이라고 가정
* 그렇기에 자기참조를 통해 실제로 하나의 bean이지만 여러 타입의 bean으로 사용되도록 구현할 예정

```java
@RequiredConstructor
public class XmlSqlService implements SqlService, SqlRegistry, SqlReader {
    private SqlReader sqlReader; // SqlService 로써의 상태 값
    private SqlRegistry sqlRegistry; // SqlService 로써의 상태 값
 
    private Map<String, String> sqlMap = new HashMap<String>(); // SqlRegistry 로써의 상태값
    private final String loadSql; // SqlReader 로써의 상태값

    // 생략
}
```
* Spring은 자기자신을 DI하는 것을 허용한다.
* 자기참조빈의 주의할 점은 언제든 별개의 오브젝트로 분리할 수 있도록 별개의 인터페이스의 리소스는
항상 해당 인터페이스의 API를 통해 접근해야한다는 것이다

> * "자기참조빈"은 흔히 사용하는 방법은 아니며, 책임이 다르다면 클래스를 구분하고 다른 오브젝트로 만드는 것이 자연스럽다.
> * 다만, 관심사가 복잡하게 얽혀있어 확장이 어렵고 변경이 취약한 상태에서 변경을 시도할 때 처음 적용해볼 수 있는 방법이다.

## 4. 디폴트 의존관계의 빈
### 4.1. 디폴트 의존관계란?
* 외부에서 DI 받지 않는 경우 기본적으로 자동 적용되는 의존관계를 말한다
```java
@Bean
public class DefaultSqlService implements SqlService {
   DefaultSqlService(){
      setSqlReader(new JaxbXmlSqlReader());
      setSqlRegstry(new JaxbXmlSqlRegistry());
   }
}
```
* 장점:
  * 개인적으로 장점이 없는 것 같다
* 단점:
   * DI되는 객체의 내부에 또 DI되어야하는 정보가 있다면 문제가 된다.
   * 다른 오브젝트를 사용해도 무조건 Bean으로 생성해버려 리소스가 낭비된다
