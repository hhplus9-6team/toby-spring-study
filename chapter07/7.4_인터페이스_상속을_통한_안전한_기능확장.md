# 7.4 인터페이스 상속을 통한 안전한 기능확장

원칙적으로 권장되진 않지만 때로는 서버가 운영 중인 상태에서 재시작하지 않고 긴급하게 애플리케이션이 사용 중인 SQL을 변경해야 할 수도 있다.

지금까지 만든 SqlService 구현 클래스들은 초기에 리소스로부터 SQL 정보를 읽어오면 이를 메모리에 두고 그대로 사용한다. SQL 매핑정보 파일을 변경했다고 해서 메모리상의 SQL 정보가 갱신되진 않는다. 애플리케이션을 새로 시작하지 않고 등록된 SQL의 내용을 변경하고 싶다면 어떻게 해야 할지 생각해보자.

---

## 7.4.1 DI와 기능의 확장

지금까지 적용해왔던 DI는 특별한 기술이라기보다는 일종의 디자인 패턴 또는 프로그래밍 모델이라는 관점에서 이해하는 것이 자연스럽다. DI의 가치를 제대로 얻으려면 먼저 DI에 적합한 오브젝트 설계가 필요하다.

### DI를 의식하는 설계

초기에 SqlService의 내부 기능을 적절한 책임과 역할에 따라 분리하고, 인터페이스를 정의해 느슨하게 연결해주고, DI를 통해 유연하게 의존관계를 지정하도록 설계했기 때문에 그 뒤의 작업은 매우 쉬워졌다. 결국 유연하고 확장 가능한 좋은 오브젝트 설계와 DI 프로그래밍 모델은 서로 상호작용을 한다.

DI를 적용하려면 최소한 두 개 이상의 의존관계를 가지고 서로 협력해 일하는 오브젝트가 필요하다. 항상 의존 오브젝트는 나중에 변경될 수 있다는 점을 염두에 둬야 한다. DI는 런타임 시에 오브젝트를 다이내믹하게 연결해서 유연한 확장을 꾀하는 게 목적이기 때문이다.

> **DI란 결국 미래를 프로그래밍하는 것이다.** 확장은 항상 미래에 일어나는 일이므로 항상 미래에 일어날 변화를 예상하고 고민해야 적합한 설계가 가능해진다.

### DI와 인터페이스 프로그래밍

DI를 제대로 만들려면 두 개의 오브젝트가 인터페이스를 통해 느슨하게 연결되어야 한다. 인터페이스를 사용하는 첫 번째 이유는 **다형성**을 얻기 위해서다. 하나의 인터페이스를 통해 여러 개의 구현을 바꿔가며 사용할 수 있게 하는 것이 DI가 추구하는 첫 번째 목적이다.

프록시, 데코레이터, 어댑터, 테스트 대역 등의 다양한 목적을 위해 인터페이스를 통한 다형성이 활용된다.

---

## 7.4.2 인터페이스 상속

하나의 오브젝트가 구현하는 인터페이스를 여러 개 만들어서 구분하는 이유 중 하나는 오브젝트의 기능이 발전하는 과정에서 다른 종류의 클라이언트가 등장하기 때문이다. 인터페이스 분리 원칙이 주는 장점은 모든 클라이언트가 자신의 관심에 따른 접근 방식을 불필요한 변경 없이 유지할 수 있다는 점이다.

### 기존 SqlRegistry 인터페이스

BaseSqlService는 SqlRegistry 인터페이스를 구현하는 오브젝트에 의존하고 있다. 현재 SqlRegistry 인터페이스는 SQL을 등록하는 기능과 등록된 SQL을 검색하는 두 가지 메소드를 갖고 있다.

```java
public interface SqlRegistry {
    void registerSql(String key, String sql);
    String findSql(String key) throws SqlNotFoundException;
}
```

### 문제: SQL 수정 기능 추가 요구

이미 등록된 SQL을 변경할 수 있는 기능을 넣어야 한다면 어떻게 해야 할까?

이때 SqlRegistry 인터페이스 자체를 수정하는 건 바람직하지 않다. 이미 SqlRegistry 인터페이스를 이용해 접근하는 클라이언트인 BaseSqlService 클래스와 그 서브클래스가 존재하기 때문이다. BaseSqlService 입장에선 SQL을 읽어오는 기능이면 충분하므로, **인터페이스 분리 원칙**을 지키기 위해서라도 이미 적용된 SqlRegistry는 건드리면 안 된다.

### 해결: 인터페이스 상속으로 확장

새롭게 추가된 기능을 사용하는 클라이언트를 위해 새로운 인터페이스를 정의한다. 기존의 SqlRegistry 인터페이스를 상속하고, SQL 수정 기능을 추가한 `UpdatableSqlRegistry` 인터페이스를 정의한다.

```java
public interface UpdatableSqlRegistry extends SqlRegistry {

    void updateSql(String key, String sql) throws SqlUpdateFailureException;

    void updateSql(Map<String, String> sqlmap) throws SqlUpdateFailureException;
}
```

- `UpdatableSqlRegistry`는 `SqlRegistry`의 서브인터페이스다.
- SQL의 등록과 조회가 가능한 SqlRegistry에 SQL 업데이트 기능을 추가했다.
- BaseSqlService는 여전히 기존 SqlRegistry 인터페이스를 통해 접근하면 충분하다.

### 의존관계 설정

SQL 변경 요청을 담당하는 `SqlAdminService`는 `UpdatableSqlRegistry` 인터페이스를 통해 SQL 레지스트리 오브젝트에 접근해야 한다.

```xml
<bean id="sqlService" class="springbook.user.sqlservice.BaseSqlService">
    ...
    <property name="sqlRegistry" ref="sqlRegistry" />
</bean>

<bean id="sqlRegistry" class="springbook.user.sqlservice.MyUpdatableSqlRegistry" />

<bean id="sqlAdminService" class="springbook.user.sqlservice.SqlAdminService">
    <property name="updatableSqlRegistry" ref="sqlRegistry" />
</bean>
```

BaseSqlService와 SqlAdminService는 **동일한 `MyUpdatableSqlRegistry` 오브젝트**를 DI 받아서 사용하지만, 각자의 관심과 필요에 따라 다른 인터페이스를 통해 접근한다.

```java
public class SqlAdminService implements AdminEventListener {

    private UpdatableSqlRegistry updatableSqlRegistry;

    public void setUpdatableSqlRegistry(UpdatableSqlRegistry updatableSqlRegistry) {
        this.updatableSqlRegistry = updatableSqlRegistry;
    }

    public void updateEventListener(UpdateEvent event) {
        this.updatableSqlRegistry.updateSql(event.get(KEY_ID), event.get(SQL_ID));
    }
}
```

### 핵심 정리

| 클라이언트 | 사용 인터페이스 | 접근 가능한 기능 |
|---|---|---|
| BaseSqlService | SqlRegistry | 등록, 조회 |
| SqlAdminService | UpdatableSqlRegistry | 등록, 조회, **수정** |

인터페이스를 추가하거나 상속을 통해 확장하는 방식을 잘 활용하면, 이미 기존 인터페이스를 사용하는 클라이언트가 있는 경우에도 유연한 확장이 가능하다. 중요한 것은 **클라이언트가 정말 필요로 하는 기능을 가진 인터페이스를 통해 오브젝트에 접근하도록 만드는 것**이다.

> 잘 적용된 DI는 결국 잘 설계된 오브젝트 의존관계에 달려 있다. 인터페이스를 적절하게 분리하고 확장하는 방법을 통해 기존의 클라이언트에 영향을 주지 않으면서 유연한 확장을 얻을 수 있다. DI와 객체지향 설계는 서로 밀접한 관계를 맺고 있다.
