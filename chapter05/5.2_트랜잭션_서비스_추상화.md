# 5.2 트랜잭션 서비스 추상화

> 트랜잭션이란 "더 이상 나눌 수 없는 단위 작업"이다. 작업을 쪼개서 부분적으로만 적용할 수 없으며, 중간에 예외가 발생하면 초기 상태로 완전히 복구되어야 한다.

## 5.2.1 모 아니면 도

`upgradeLevels()` 메소드가 사용자 레벨을 업그레이드하는 도중 예외가 발생하면, 이미 변경된 사용자는 그대로 유지되고 나머지는 변경되지 않는 문제가 발생한다. 예를 들어 100명 중 23명까지 업그레이드된 후 장애가 발생하면, 23명만 변경된 비정상 상태가 된다.

### 테스트용 UserService 대역

기존 코드를 수정하지 않고 예외 상황을 재현하기 위해 **상속**을 이용한다. `upgradeLevel()` 메소드를 오버라이드하여 특정 사용자 처리 시 예외를 강제로 발생시킨다.

```java
static class TestUserService extends UserService {
    private String id;

    private TestUserService(String id) { // 예외발생시킬 User의 id 지정 가능
        this.id = id;
    }

    protected void upgradeLevel(User user) { // 특정 User의 id와 일치하면 예외 발생
        if (user.getId().equals(this.id)) throw new TestUserServiceException();
        super.upgradeLevel(user);
    }
}
```

### 강제 예외 발생을 통한 테스트

```java
@Test
public void upgradeAllOrNothing() {
    UserService testUserService = new TestUserService(users.get(3).getId());
    testUserService.setUserDao(this.userDao); // 수동 DI

    userDao.deleteAll();
    for (User user : users) userDao.add(user);

    try {
        testUserService.upgradeLevels();
        fail("TestUserServiceException expected");
    } catch (TestUserServiceException e) {
    }

    checkLevelUpgraded(users.get(1), false);  // 레벨 변경이 취소되었는지 확인
}
```

이 테스트는 **실패**한다. 예외가 발생하기 전에 이미 업그레이드된 사용자의 레벨이 원래대로 돌아가지 않았다.  
upgradeLevels() 메소드가 하나의 트랜잭션 안에서 동작하지 않았기에.. 죽, 트랜잭션 경계설정 코드가 존재하지 않아 각 사용자 업데이트가 독립적인 트랜잭션으로 처리되었기 떄문이다.

## 5.2.2 트랜잭션 경계설정

하나의 SQL 명령을 처리하는 경우는 DB가 트랜잭션을 보장한다고 믿을 수 있다. 하지만 여러 SQL 명령을 하나의 트랜잭션으로 묶어서 처리하려면, 트랜잭션 경계설정이 필요하다.

- **트랜잭션 롤백(transaction rollback)**: 작업 중간에 예외가 발생하면, 앞에서 처리한 SQL 작업도 모두 취소하고 트랜잭션 시작 이전의 상태로 되돌리는 것이다.
- **트랜잭션 커밋(transaction commit)**: 모든 SQL 작업이 성공적으로 마무리되었다고 DB에 알려서 변경사항을 확정하는 것이다.


### JDBC 트랜잭션의 경계설정

```java
Connection c = dataSource.getConnection();

c.setAutoCommit(false);  // 트랜잭션 시작
try {
    PreparedStatement st1 = c.prepareStatement("update users ...");
    st1.executeUpdate();

    PreparedStatement st2 = c.prepareStatement("delete users ...");
    st2.executeUpdate();

    c.commit();    // 트랜잭션 커밋
} catch (Exception e) {
    c.rollback();  // 트랜잭션 롤백
}

c.close();
```

- **트랜잭션 경계설정(transaction demarcation)**: `setAutoCommit(false)`로 시작하여 `commit()` 또는 `rollback()`으로 끝나는 경계
- **로컬 트랜잭션(local transaction)**: 하나의 DB Connection 안에서 만들어지는 트랜잭션

### UserService와 UserDao의 트랜잭션 문제

JdbcTemplate의 메소드는 각각 독립적인 트랜잭션으로 실행된다. 하나의 템플릿 메소드 호출이 한 개의 트랜잭션이므로, `upgradeLevels()` 안에서 여러 번 호출되는 `userDao.update()`는 각각 별개의 트랜잭션으로 처리된다.  
모든 사용자의 레벨을 업그레이드하는 작업인 `upgradeLevels()` 메소드의 경우, 모든 사용자의 레벨 업그레이드 작업은 새로 시작된 하나의 트랜잭션 안에서 진행돼야 한다.

### 비즈니스 로직 내의 트랜잭션 경계설정

`upgradeLevels()`와 같이 여러 SQL을 사용하는 작업을 하나의 트랜잭션으로 묶으려면 트랜잭션의 경계설정 작업을 UserService 쪽으로 가져와야 한다.

### Connection 파라미터 전달 방식의 문제점

UserService에서 Connection을 생성하고 DAO 메소드에 파라미터로 전달하는 방식은 다음 문제를 일으킨다.

1. **JdbcTemplate 활용 불가**: JDBC API를 직접 사용해야 하고 try/catch/finally 코드가 복잡해진다
2. **DAO 메소드에 Connection 파라미터 추가**: UserDao 인터페이스의 모든 메소드에 Connection을 전달해야 한다
3. **기술 독립성 상실**: Connection은 JDBC 전용이므로 JPA나 Hibernate로 전환할 때 UserDao 인터페이스부터 바꿔야 한다
4. **테스트 코드 영향**: 테스트에서도 Connection 오브젝트를 일일이 만들어서 DAO 메소드에 전달해야 한다

## 5.2.3 트랜잭션 동기화

### 트랜잭션 동기화란?

UserService에서 트랜잭션을 시작하기 위해 만든 Connection 오브젝트를 **특별한 저장소에 보관**해두고, 이후에 호출되는 DAO의 메소드에서는 저장된 Connection을 가져다가 사용하게 하는 것이다.

### 트랜잭션 동기화 적용

```java
private DataSource dataSource;

public void setDataSource(DataSource dataSource) {
    this.dataSource = dataSource;
}

public void upgradeLevels() throws Exception {
    // 트랜잭션 동기화 관리자를 이용해 동기화 작업 초기화
    TransactionSynchronizationManager.initSynchronization();
    // DB 커넥션 생성과 동시에 트랜잭션 동기화에 사용하도록 저장소에 바인딩
    Connection c = DataSourceUtils.getConnection(dataSource);
    c.setAutoCommit(false);

    try {
        List<User> users = userDao.getAll();
        for (User user : users) {
            if (canUpgradeLevel(user)) {
                upgradeLevel(user);  // Connection 파라미터 전달 불필요
            }
        }
        c.commit();
    } catch (Exception e) {
        c.rollback();
        throw e;
    } finally {
        DataSourceUtils.releaseConnection(c, dataSource);  // DB 커넥션 닫기
        // 동기화 작업 종료 및 정리
        TransactionSynchronizationManager.unbindResource(this.dataSource);
        TransactionSynchronizationManager.clearSynchronization();
    }
}
```

`DataSourceUtils.getConnection()`을 사용하는 이유는, Connection 오브젝트를 생성해줄 뿐만 아니라 **트랜잭션 동기화에 사용하도록 저장소에 바인딩**해주기 때문이다.

### 트랜잭션 동기화의 동작 흐름

1. UserService에서 Connection을 생성하고 트랜잭션을 시작
2. 생성된 Connection을 트랜잭션 동기화 저장소에 저장
3. 이후 JdbcTemplate 메소드에서는 트랜잭션 동기화 저장소에 현재 시작된 트랜잭션의 Connection이 있는지 확인
4. 저장소에서 Connection을 가져와 사용 (같은 트랜잭션에 참여)
5. 모든 작업이 완료되면 commit() 또는 rollback() 호출
6. 트랜잭션 저장소에서 Connection 오브젝트를 제거

### 트랜잭션 동기화의 장점

- Connection 파라미터 전달 없이 트랜잭션 보장
- JdbcTemplate을 그대로 사용 가능
- UserDao 인터페이스 변경 불필요
- 멀티스레드 환경에서 안전 (스레드마다 독립적인 Connection을 관리)

## 5.2.4 트랜잭션 서비스 추상화

### 기술과 환경에 종속되는 트랜잭션 경계설정 코드

#### 로컬 트랜잭션의 한계

로컬 트랜잭션은 하나의 DB Connection에 종속되기 때문에, 여러 개의 DB에 데이터를 넣는 작업을 하나의 트랜잭션으로 만들 수 없다.

#### 글로벌/분산 트랜잭션

별도의 트랜잭션 관리자를 통해 여러 DB의 트랜잭션을 관리하는 **글로벌 트랜잭션** 방식을 사용해야 한다. 자바에서는 JTA(Java Transaction API)를 제공하고 있다.

```java
// JTA를 이용한 트랜잭션 코드 구조
InitialContext ctx = new InitialContext();
UserTransaction tx = (UserTransaction) ctx.lookup(USER_TX_JNDI_NAME);

tx.begin();
Connection c = dataSource.getConnection();  // JNDI로 가져온 DataSource 사용
try {
    // 데이터 액세스 코드
    tx.commit();
} catch (Exception e) {
    tx.rollback();
    throw e;
} finally {
    c.close();
}
```

#### 기술 종속 문제

JDBC 로컬 트랜잭션을 JTA로 바꾸려면 UserService의 코드를 수정해야 한다. 하이버네이트를 이용하는 경우에도 Connection 대신 Session을 사용하는 독자적인 트랜잭션 관리 API를 사용해야 한다. **기술이 바뀔 때마다 비즈니스 로직이 담긴 UserService 코드가 함께 변경**되는 것이 핵심 문제이다.

### 트랜잭션 API의 의존관계 문제와 해결책

UserService가 특정 트랜잭션 방식에 의존하지 않으려면 **추상화**를 도입해야 한다.

> 추상화란 하위 시스템의 공통점을 뽑아내서 분리시키는 것이다.

JDBC, JTA, 하이버네이트, JPA 등 모두 트랜잭션의 개념(시작, 커밋, 롤백)을 갖고 있으므로, 이 공통점을 모아서 추상화된 트랜잭션 관리 계층을 만들 수 있다.

### 스프링의 트랜잭션 서비스 추상화

스프링은 **PlatformTransactionManager** 인터페이스를 통해 트랜잭션 기술의 공통점을 담은 추상화 계층을 제공한다.

```java
public interface PlatformTransactionManager {
    TransactionStatus getTransaction(TransactionDefinition definition);
    void commit(TransactionStatus status);
    void rollback(TransactionStatus status);
}
```

#### 구현 클래스

| 구현 클래스 | 용도 |
|---|---|
| `DataSourceTransactionManager` | JDBC 로컬 트랜잭션 |
| `JtaTransactionManager` | JTA 글로벌 트랜잭션 |
| `HibernateTransactionManager` | Hibernate 트랜잭션 |
| `JpaTransactionManager` | JPA 트랜잭션 |

#### 트랜잭션 추상화 적용 코드

```java
public void upgradeLevels() {
    PlatformTransactionManager transactionManager =
        new DataSourceTransactionManager(dataSource);

    TransactionStatus status =
        transactionManager.getTransaction(new DefaultTransactionDefinition());

    try {
        List<User> users = userDao.getAll();
        for (User user : users) {
            if (canUpgradeLevel(user)) {
                upgradeLevel(user);
            }
        }
        transactionManager.commit(status);
    } catch (RuntimeException e) {
        transactionManager.rollback(status);
        throw e;
    }
}
```

- `getTransaction()`: 트랜잭션을 시작하면서 필요한 커넥션을 자동으로 가져온다
- `DefaultTransactionDefinition`: 트랜잭션에 대한 기본 속성을 담고 있다
- `TransactionStatus`: 시작된 트랜잭션의 조작이 필요할 때 `commit()`이나 `rollback()`에 전달하는 파라미터

### 트랜잭션 기술 설정의 분리

위 코드에서 `new DataSourceTransactionManager(dataSource)`처럼 구체적인 클래스를 직접 생성하면 DI 원칙에 어긋난다. 어떤 트랜잭션 매니저 구현 클래스를 사용할지는 **컨테이너를 통해 외부에서 제공**받게 해야 한다.

```java
public class UserService {
    private PlatformTransactionManager transactionManager;

    public void setTransactionManager(PlatformTransactionManager transactionManager) {
        this.transactionManager = transactionManager;
    }

    public void upgradeLevels() {
        TransactionStatus status =
            this.transactionManager.getTransaction(new DefaultTransactionDefinition());

        try {
            List<User> users = userDao.getAll();
            for (User user : users) {
                if (canUpgradeLevel(user)) {
                    upgradeLevel(user);
                }
            }
            this.transactionManager.commit(status);
        } catch (RuntimeException e) {
            this.transactionManager.rollback(status);
            throw e;
        }
    }
}
```

#### 스프링 빈 설정

```xml
<bean id="userService" class="springbook.user.service.UserService">
    <property name="userDao" ref="userDao" />
    <property name="transactionManager" ref="transactionManager" />
</bean>

<bean id="transactionManager"
      class="org.springframework.jdbc.datasource.DataSourceTransactionManager">
    <property name="dataSource" ref="dataSource" />
</bean>
```

기술을 변경할 때는 **빈 설정만 수정**하면 되고 UserService 코드는 전혀 수정할 필요가 없다.

```xml
<!-- JTA로 변경할 경우 -->
<bean id="transactionManager"
      class="org.springframework.transaction.jta.JtaTransactionManager" />

<!-- Hibernate로 변경할 경우 -->
<bean id="transactionManager"
      class="org.springframework.orm.hibernate3.HibernateTransactionManager">
    <property name="sessionFactory" ref="sessionFactory" />
</bean>
```
