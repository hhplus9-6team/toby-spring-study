# 5.4 메일 서비스 추상화


## 5.4.1 JavaMail을 이용한 메일 발송 기능

### User에 이메일 필드 추가

레벨 업그레이드 시 안내 메일을 보내려면 User에 email 필드를 추가해야 한다.

```java
public class User {
    // ...
    String email;

    public String getEmail() { return email; }
    public void setEmail(String email) { this.email = email; }
}
```

User 테이블에도 email 컬럼을 추가하고, UserDao의 관련 메소드들(add, get, update 등)과 테스트도 수정한다.

### JavaMail 메일 발송

자바에서 메일을 발송할 때는 표준 기술인 **JavaMail**을 사용한다.

```java
private void sendUpgradeEMail(User user) {
    Properties props = new Properties();
    props.put("mail.smtp.host", "mail.ksug.org");
    Session s = Session.getInstance(props, null);

    MimeMessage message = new MimeMessage(s);
    try {
        message.setFrom(new InternetAddress("useradmin@ksug.org"));
        message.addRecipient(Message.RecipientType.TO,
            new InternetAddress(user.getEmail()));
        message.setSubject("Upgrade 안내");
        message.setText("사용자님의 등급이 " + user.getLevel().name() + "로 업그레이드되었습니다.");

        Transport.send(message);
    } catch (AddressException e) {
        throw new RuntimeException(e);
    } catch (MessagingException e) {
        throw new RuntimeException(e);
    }
}
```

이 메소드를 `upgradeLevel()` 메소드에서 호출하도록 추가한다.

```java
protected void upgradeLevel(User user) {
    user.upgradeLevel();
    userDao.update(user);
    sendUpgradeEMail(user);
}
```

## 5.4.2 JavaMail이 포함된 코드의 테스트

### 메일 발송 테스트의 문제점

메일 발송 기능이 포함된 코드를 테스트할 때 발생하는 문제:

1. **운영 메일 서버에 부담**: 테스트를 실행할 때마다 실제 메일이 전송된다
2. **메일 서버 의존**: 테스트 환경에 메일 서버가 없으면 테스트 자체가 실패한다
3. **불필요한 메일 수신**: 테스트 때마다 메일을 실제로 받는 것은 바람직하지 않다

### 테스트용 메일 서버

테스트용 메일 서버를 사용하여 외부로 실제 메일을 발송하지 않고 요청만 받도록 할 수 있다. 하지만 여전히 JavaMail 코드가 실행되면서 외부 시스템(메일 서버)에 의존하는 문제가 남는다.

### 더 나은 접근

JavaMail을 통해 메일 서버까지만 메일이 잘 전달되면, 결국 사용자에게도 메일이 잘 보내졌다고 생각할 수 있다. 그렇다면 **JavaMail을 통해 메일 서버에 요청이 들어간다는 보장만 있으면** 굳이 테스트 때마다 JavaMail을 직접 구동시킬 필요가 없다.

## 5.4.3 테스트를 위한 서비스 추상화

### JavaMail의 문제: 테스트하기 어려운 구조

JavaMail의 핵심 API는 테스트하기 매우 어려운 구조를 가지고 있다.

- `Session` 오브젝트는 인터페이스가 아닌 **구체 클래스**이고, 생성자가 모두 private이며 **static 팩토리 메소드**만 제공한다
- `Session` 클래스는 **상속이 불가능한 final 클래스**다
- `Transport`도 마찬가지로 **static 메소드**로 메일을 전송한다

결국 JavaMail은 DataSource처럼 테스트용 오브젝트로 바꿔치기하는 것이 불가능하다.

### 스프링의 메일 서비스 추상화

스프링은 JavaMail에 대한 추상화 기능을 제공한다. **MailSender** 인터페이스가 그 핵심이다.

```java
public interface MailSender {
    void send(SimpleMailMessage simpleMessage) throws MailException;
    void send(SimpleMailMessage[] simpleMessages) throws MailException;
}
```

스프링이 제공하는 `JavaMailSenderImpl` 클래스는 내부적으로 JavaMail API를 사용하여 메일을 발송하는 MailSender 구현체다.

### MailSender 적용

UserService에 MailSender 인터페이스를 DI 받도록 수정한다.

```java
public class UserService {
    private MailSender mailSender;

    public void setMailSender(MailSender mailSender) {
        this.mailSender = mailSender;
    }

    private void sendUpgradeEMail(User user) {
        SimpleMailMessage mailMessage = new SimpleMailMessage();
        mailMessage.setTo(user.getEmail());
        mailMessage.setFrom("useradmin@ksug.org");
        mailMessage.setSubject("Upgrade 안내");
        mailMessage.setText("사용자님의 등급이 " + user.getLevel().name() + "로 업그레이드되었습니다.");

        this.mailSender.send(mailMessage);
    }
}
```

스프링 설정에서 JavaMailSenderImpl을 빈으로 등록한다.

```xml
<bean id="mailSender" class="org.springframework.mail.javamail.JavaMailSenderImpl">
    <property name="host" value="mail.server.com" />
</bean>

<bean id="userService" class="springbook.user.service.UserService">
    <property name="userDao" ref="userDao" />
    <property name="transactionManager" ref="transactionManager" />
    <property name="mailSender" ref="mailSender" />
</bean>
```

### 테스트용 메일 발송 오브젝트: DummyMailSender

테스트 환경에서는 실제 메일을 보낼 필요가 없으므로, 아무 기능도 하지 않는 **DummyMailSender**를 만든다.

```java
public class DummyMailSender implements MailSender {
    public void send(SimpleMailMessage mailMessage) throws MailException {
    }

    public void send(SimpleMailMessage[] mailMessages) throws MailException {
    }
}
```

테스트 설정에서는 JavaMailSenderImpl 대신 DummyMailSender를 빈으로 등록하면, UserService 코드를 전혀 수정하지 않고도 테스트 시에는 메일이 발송되지 않도록 할 수 있다.

```xml
<bean id="mailSender" class="springbook.user.service.DummyMailSender" />
```

### 서비스 추상화의 의미

일반적으로 서비스 추상화라고 하면, 트랜잭션과 같이 기능은 유사하나 사용 방법이 다른 로우레벨의 다양한 기술에 대해 **추상 인터페이스와 일관성 있는 접근 방법을 제공**해주는 것을 말한다.

반면 JavaMail의 경우처럼, 테스트를 어렵게 만드는 건전하지 않은 방식으로 설계된 API를 사용해야 하는 경우에도 서비스 추상화는 유용하다. JavaMail처럼 **확장이 불가능하게 설계해놓은 API를 사용해야 하는 경우라면 추상화 계층의 도입을 적극 고려**해볼 필요가 있다.

## 5.4.4 테스트 대역

### 테스트 대역(Test Double)의 개념

테스트 환경을 만들어주기 위해, 테스트 대상이 되는 오브젝트의 기능에만 충실하게 수행하면서 **빠르게, 자주 테스트를 실행**할 수 있도록 사용하는 오브젝트를 통틀어 **테스트 대역**(test double)이라고 한다.

DummyMailSender도 UserServiceTest의 메일 전송 기능에 대한 테스트 대역이다.

### 테스트 스텁(Test Stub)

테스트 스텁은 테스트 대상 오브젝트의 의존 객체로서 존재하면서, 테스트 동안에 코드가 **정상적으로 수행할 수 있도록 돕는** 것을 말한다.

- DummyMailSender처럼 호출만 하고 아무 일도 하지 않는 경우
- 리턴 값이 필요한 경우 미리 준비한 값을 반환하도록 구성
- 강제로 예외를 발생시키는 경우

### 목 오브젝트(Mock Object)

테스트 대상 오브젝트가 의존 오브젝트에게 **출력한 값에 관심이 있는 경우**가 있다. 또는 의존 오브젝트를 **얼마나 사용했는가** 하는 커뮤니케이션 행위 자체에 관심이 있을 수 있다.

목 오브젝트는 스텁처럼 테스트 오브젝트가 정상적으로 실행되도록 도와주면서, 테스트 오브젝트와 자신 사이에서 **일어나는 커뮤니케이션 내용을 저장**해뒀다가 테스트 결과를 **검증하는 데 활용**할 수 있게 해준다.

### 목 오브젝트를 이용한 테스트

`upgradeLevels()` 테스트에서 실제로 메일 발송 요청이 제대로 이뤄졌는지 검증하고 싶다면, 목 오브젝트를 만들어 사용한다.

```java
static class MockMailSender implements MailSender {
    private List<String> requests = new ArrayList<String>();

    public List<String> getRequests() {
        return requests;
    }

    public void send(SimpleMailMessage mailMessage) throws MailException {
        requests.add(mailMessage.getTo()[0]);  // 수신자 메일 주소를 저장
    }

    public void send(SimpleMailMessage[] mailMessages) throws MailException {
    }
}
```

### 목 오브젝트를 활용한 테스트 코드

```java
@Test
public void upgradeLevels() throws Exception {
    userDao.deleteAll();
    for (User user : users) userDao.add(user);

    MockMailSender mockMailSender = new MockMailSender();
    userService.setMailSender(mockMailSender);

    userService.upgradeLevels();

    // 레벨 업그레이드 검증
    checkLevelUpgraded(users.get(0), false);
    checkLevelUpgraded(users.get(1), true);
    checkLevelUpgraded(users.get(2), false);
    checkLevelUpgraded(users.get(3), true);
    checkLevelUpgraded(users.get(4), false);

    // 목 오브젝트를 이용한 메일 발송 검증
    List<String> request = mockMailSender.getRequests();
    assertThat(request.size(), is(2));
    assertThat(request.get(0), is(users.get(1).getEmail()));
    assertThat(request.get(1), is(users.get(3).getEmail()));
}
```

목 오브젝트를 통해 **누구에게 메일을 보냈는지**, **몇 번 보냈는지**까지 검증할 수 있다. 보통의 테스트 방법으로는 검증하기가 매우 까다로운 테스트 대상 오브젝트의 **내부에서 일어나는 일이나 다른 오브젝트 사이에서 주고받는 정보까지 검증**하는 일이 목 오브젝트를 사용하면 간편해진다.

## 정리

서비스 추상화는 기술 변경 대응뿐 아니라, JavaMail처럼 **테스트하기 어려운 API를 테스트 가능하게 만드는 것**만으로도 충분한 가치가 있다. DI를 통해 의존 오브젝트를 테스트 대역으로 손쉽게 교체할 수 있으므로, 테스트 가능한 설계를 위해 **추상화 계층의 도입을 적극 고려**해야 한다.
