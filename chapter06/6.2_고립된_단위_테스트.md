# 6.2 고립된 단위 테스트

## 1. 왜 고립된 테스트가 필요한가

앞 장에서 `UserService` 테스트를 작성하면서 우리는 다음과 같은 문제를 발견합니다.

- 테스트가 DB에 의존
- 테스트 속도가 느림
- 테스트 겨과가 환경에 영향을 받음

즉, 테스트가 외부 의존성 (DB, 트랜잭션 등)에 묶여 있습니다.

이런 테스트는 진짜로 검증하고 싶은 것보다 환경에 더 많이 의존하게 됩니다.

정말 검증해야 하는 것은 **`UserService` 의 비즈니스 로직**입니다. 

이 로직을 외부 환경 없이 검증한느 것이 바로 고립된 단위 테스트입니다.

## 2. 단위 테스트와 통합 테스트

테스트는 보통 **검증 범위**에 따라 다음 두 가지로 나눌 수 있습니다.

### 1) 단위 테스트

- 하나의 클래스나 메서드처럼 작은 단위를 검증
- 외부 의존성은 Mock, Stub, Fake 등으로 대체
- 빠르고 실패 원인이 명확
- 예시 : `UserService` 의 비즈니스 규칙만 검증

### 2) 통합 테스트

- 여러 컴포넌트를 함께 테스트
- 스프링 컨테이너, DB, 외부 시스템 등을 포함
- 실제 환경과 가깝지만 느리고 원인 파악이 어려움
- 예시 : `UserService + UserDao + DB` 까지 검증

## 3. `upgradeLevels()`의 진짜 테스트 대상

```java
public void upgradeLevels() {
    List<User> users = userDao.getAll();

    for (User user : users) {
        if (canUpgradeLevel(user)) {
            upgradeLevel(user);
        }
    }
}

private void upgradeLevel(User user) {
    user.upgradeLevel();     // 등급 변경
    userDao.update(user);    // DB 업데이트
    sendUpgradeEmail(user);  // 메일 발송
}
```

`upgradeLevels()`의 핵심 로직은 다음과 같습니다.

- 사용자 등급 판단
- 등급 업그레이드
- 사용자 정보 저장
- 메일 발송

외부 의존성이 두 가지 존재합니다.

- `UserDao`
- `MailSender`

이 두 가지를 대체할 수 있다면 `UserService`를 완전히 고립된 상태로 테스트할 수 있습니다.

기존 테스트는 실제 실행 환경과 비슷하지만, 다음 문제가 있습니다.

- DB 준비가 필요
- 속도가 느림
- 메일 발송까지 섞이면 테스트 범위가 커짐
- 실패했을 때 Service 로직 문제인지, DAO 문제인지, 설정 문제인지 바로 알기 어려움

## 5. 테스트 대역 (Test Double) 만들기

```java
public class MockUserDao implements UserDao {

    private List<User> users;
    private List<User> updated = new ArrayList<>();

    public MockUserDao(List<User> users) {
        this.users = users;
    }

    public List<User> getAll() {
        return this.users;
    }

    public void update(User user) {
        updated.add(user);
    }

    public List<User> getUpdated() {
        return updated;
    }
}
```

먼저 `UserDao`를 대신할 테스트용 클래스를 만듭니다.

## 6. 고립된 단위 테스트

```java
@Test
public void upgradeLevels() {

    List<User> users = Arrays.asList(
        new User("u1", "A", "p1", Level.BASIC, 49, 0),
        new User("u2", "B", "p2", Level.BASIC, 50, 0),
        new User("u3", "C", "p3", Level.SILVER, 60, 29),
        new User("u4", "D", "p4", Level.SILVER, 60, 30)
    );

    MockUserDao mockUserDao = new MockUserDao(users);
    MockMailSender mockMailSender = new MockMailSender();

    UserServiceImpl userService = new UserServiceImpl();
    userService.setUserDao(mockUserDao);
    userService.setMailSender(mockMailSender);

    userService.upgradeLevels();

    List<User> updated = mockUserDao.getUpdated();

    assertThat(updated.get(0).getLevel()).isEqualTo(Level.SILVER);
    assertThat(updated.get(1).getLevel()).isEqualTo(Level.GOLD);

    assertThat(mockMailSender.getRequests().size()).isEqualTo(2);
}
```

`UserService`만 실제 객체로 두고 테스트를 진행합니다.

- 스프링 컨테이너 없이 실행되고
- DB 접근 없이 동작하며
- 메일 서버 호출도 없습니다.

대신 **비즈니스 로직만 정확히 검증**합니다.

## 7. 직접 만든 Mock의 한계

하지만 테스트 대역을 직접 구현하면 문제가 발생합니다.

- 테스트용 클래스가 계속 늘어남
- 구현 코드가 반복됨
- 단순한 검증에도 많은 코드가 필요함

우리가 확인하려는 것은 사실 다음뿐입니다.

- `getAll()`이 어떤 데이터를 반환하는지
- `update()`가 몇 번 호출됐는지
- `send()`가 호출됐는지

이런 단순한 검증을 위해 클래스를 직접 만드는 일은 번거롭습니다.

## 8. Mockito로 개선하기

Mockito는 **Mock 객체를 자동으로 생성해주는 라이브러리**입니다.

### 1) Mock 생성

```java
UserDao mockUserDao = mock(UserDao.class);
```

### 2) 동작 정의

```java
when(mockUserDao.getAll()).thenReturn(users);
```

### 3) 행동 검증

```java
verify(mockUserDao, times(2)).update(any(User.class));
```

## 9. Mockito를 사용한 테스트

```java
@Test
public void upgradeLevels() {

    UserDao mockUserDao = mock(UserDao.class);
    when(mockUserDao.getAll()).thenReturn(users);

    MailSender mockMailSender = mock(MailSender.class);

    UserServiceImpl userService = new UserServiceImpl();
    userService.setUserDao(mockUserDao);
    userService.setMailSender(mockMailSender);

    userService.upgradeLevels();

    verify(mockUserDao, times(2)).update(any(User.class));
    verify(mockMailSender, times(2)).send(any(SimpleMailMessage.class));
}
```

Mockito를 사용하면

- 테스트 코드가 훨씬 간결해지고
- 필요한 동작만 선언적으로 정의할 수 있습니다.

## 9. 결론

> 고립된 단위 테스트는 외부 의존성을 제거하고 비즈니스 로직만 검증하는 테스트 방식이며,\n
> Mockito는 이를 간결하게 작성할 수 있도록 도와주는 Mock 라이브러리다.

좋은 테스트는 다음 조건을 만족해야 합니다.

- 빠르게 실행
- 외부 환경에 의존하지 않음
- 실패 원인이 명확

이를 위해 테스트 대상은 **가능한 한 작게 고립**되어야 합니다.

Mockito는 이러한 고립된 단위 테스트를 더 간결하게 작성하도록 도와주는 도구입니다.
