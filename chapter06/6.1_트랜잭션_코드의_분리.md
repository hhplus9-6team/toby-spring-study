# 6.1 트랜잭션 코드의 분리

앞서 5장에서 서비스 추상화 기법과 DI를 활용하여 트랜잭션 기술을 독립적으로 분리하였습니다.

```java
public void upgradeLebels() throw Exception {
 TransactionStatus status = this.transactionManager.getTransaction(new DefaultTransactionDefinition());

 try {
  // do something
  this.transactionManger.commit(status);
 } catch (Exception e) {
  this.transactionManger.rollback(status);
 }
}
```

하지만 위 코드 역시 비지니스 로직의 주인인 사용자 등급 변경보다 트랜잭션 관리 코드가 더 많은 부분을 차지하고 있다는 문제점이 있습니다.
트랜잭션의 경계는 비지니스 로직 전후에 설정되어야 하는 것은 분명하기 때문에 UserService에서 트랜잭션을 관리하는 것은 합리적입니다.

## 6.1.1 메소드 분리

위 코드를 살펴보면 1)트랜잭션 경계 설정과 2)비지니스 로직이 뚜렷하게 분리되어있고,
트랜잭션 경계 설정 코드와 비지니스 로직 코드간에 서로 주고받는 정보가 없다는 것을 확인할 수 있습니다.

따라서 이 두가지 코드는 완벽하게 독립적인 코드입니다.

```java
//성격이 다른 두 코드를 메소드 분리
public void upgradeLebels() throw Exception {
 TransactionStatus status = this.transactionManager.getTransaction(new DefaultTransactionDefinition());

 try {
  upgradeLevelsInternal();
  this.transactionManger.commit(status);
 } catch (Exception e) {
  this.transactionManger.rollback(status);
 }
}

private void upgradeLevelsInternal() {
 List<User> users = userDao.getAll();
 for (User user : users) {
  if (canUpgradeLebel(user)) {
   upgradeLevel(user);
  }
 }
}
```

## 6.1.2 DI를 이용한 클래스의 분리

비지니스 로직과 트랜잭션 경계 코드가 분리되어 보기에는 좋아졌으나 여전히 UserService에 트랜잭션을 담당하는 기술적인 코드가 작성되어있습니다.
이를 클래스 밖으로 뽑아내는 방식으로 개선해보겠습니다.

### DI 적용을 이용한 트랜잭션 분리

현재 UserService를 사용하고있는 클라이언트는 UserService 클래스 자체를 사용한다.
따라서 트랜잭션 코드가 UserService 밖으로 분리가 될 경우 클라이언트 코드에서는 트랜잭션 기능이 빠진 UserService를 사용할 수 밖에 없다.
--> 구체적인 구현 클래스를 사용할 때 발생하는 전형적인 단점 (책임을 분리하는 과정에서 기능이 유실)

우선은 UserService를 인터페이스로 만들고 클라이언트(UserServiceTest, Controller)에서는 인터페이스를 참조하도록 결합을 제거해준다.
`Client(UserServiceTest)` -----> `<<interface>>UserService` <----- `UserServiceImpl`

인터페이스를 활용하여 런타임 시 DI를 적용하는 방법을 쓰는 이유는 일반적으로 환경(테스트, 운영)에 따라 구현 클래스를 바꿔가면서 사용하기 위함이다.
하지만 한 번에 두개의 인터페이스 구현 클래스를 동시에 사용할 수도 있다.
비지니스 로직 수행을 담당하는 UserService 인터페이스 구현체와, 트랜잭션을 담당하는 구현체 두 개를 작성하여 트랜잭션 담당 구현체가 실제 비지니스 로직 수행을
비지니스 수행 구현체에게 위임하는 방식을 고려해 볼 수 있다.

`Client(UserServiceTest)` -----> `<<interface>>UserService` <----- `UserServiceImpl(비지니스 로직), UserServiceTx(트랜잭션 경계)`

### UserService 인터페이스 도입

기존의 UserService 클래스를 인터페이스로 분리하고, 비지니스 로직을 수행할 UserServiceImpl 구현체를 작성하겠습니다.

```java
public interface UserService {
 void add(User user);
 void upgradeLevels();
}

//트랜잭션 경계를 고려하지 않고 순수하게 비지니스 로직 수행 역할의 UserServiceImpl 클래스
public class UserServiceImpl implements UserService {
 UserDao userDao;
 MailSender mailSender;

 public void add(User user) {
  //...
 }

 public void upgradeLevels() {
  List<User> users = userDao.getAll();
  for (User user : users) {
   if (canUpgradeLebel(user)) {
    upgradeLevel(user);
   }
  }
 }
}
```

### 분리된 트랜잭션 기능

비지니스 트랜잭션 처리를 담은 UserServiceTx 구현체를 작성하겠습니다.

```java
public class UserServiceTx implements UserService {
 UserService userService; //실제 비지니스 로직 수행을 위한 UserService
 PlatformTransactionManager platformTransactionManager; //트랜잭션 경계를 담당

 public void setTransactionManager(TransactionManager transactionManager) {
  this.transactionManager = transactionManager;
 }

 public void setUserService(UserService userService) {
  this.userService = userService;
 }

 public void add(User user) {
  userService.add(user);
 }

 public void upgradeLevels() {
  TransactionStatus status = this.transactionManager.getTransaction(new DefaultTransactionDefinition());

  try {

   userService.upgradeLevels();
   this.transactionManager.commit(status);
  } catch (Exception e) {
   this.transactionManager.rollback(status);
  }
 }
}
```

### 트랜잭션 적용을 위한 DI 설정

이제 UserService의 기능을 실행하기 위한 클래스의 의존 관계는 `Client` -> `UserServiceTx` -> `UserServiceImpl` 와 같습니다.
프로퍼티 정보에 이를 위한 설정 정보를 변경해야합니다.

```xml
<bean id="userService" class="user.service.UserServiceTx">
 <property name="transactionManager" ref="transactionManager"></property>
 <property name="userService" ref="userServiceImpl"></property>
</bean>

<bean id="userServiceImpl" class="user.service.UserServiceImpl">
 <property name="userDao" ref="userDao"></property>
 <property name="mailSender" ref="mailSender"></property>
</bean>
```

### 트랜잭션 분리에 따른 테스트 수정

UserService 클래스를 그대로 사용하는 UserServiceTest 코드 역시 DI를 적용한 형태로 수정이 필요하다.
`@Autowired`는 기본적으로 타입이 일치하는 빈을 찾아 주입받기 때문에 인터페이스의 구현체가 2개 이상일 경우 필드의 이름을 사용하여 적절한 빈을 찾는다.
`@Autowired UserService userService` (UserServiceTx 사용), `@Autowired UserService userServiceImpl` (UserServiceImpl 사용)

### 트랜잭션 경계설정 코드 분리의 장점

트랜잭션 경계 설정 코드를 분리함으로 얻을 수 있는 장점은

1. 비지니스 로직을 작성하는 UserServiceImpl의 코드를 작성할 때 트랜잭션과 같은 기술적인 내용을 신경쓰지 않고 비지니스 로직에 집중할 수 있다.
2. 비지니스 로직에 대한 테스트를 손쉽게 만들 수 있다.
