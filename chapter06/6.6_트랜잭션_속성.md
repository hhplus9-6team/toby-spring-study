# 6.6 트랜잭션 속성

앞선 장에서는 트랜잭션을 적용하는 방법과 AOP 기반 트랜잭션 관리 방식을 살펴보았습니다.

하지만 실제 애플리케이션에서는 단순히 트랜잭션을 시작하고 종료하는 것만으로는 부족합니다.


어떤 경우에는 기존 트랜잭션에 참여해야 하고,

어떤 경우에는 새로운 트랜잭션을 시작해야 하며,

조회 작업에서는 읽기 전용 트랜잭션을 사용하는 것이 더 적절할 수도 있습니다.


이처럼 **트랜잭션의 동작 방식을 정의하는 설정을 트랜잭션 속성(Transaction Attribute)** 이라고 합니다.


## 6.6.1 트랜잭션 정의

스프링에서 트랜잭션 정의는 다음과 같은 속성들로 구성됩니다.

- 트랜잭션 전파 (Propagation)

- 격리 수준 (Isolation)

- 제한 시간 (Timeout)

- 읽기 전용 (ReadOnly)

이 속성들은 `TransactionDefinition` 인터페이스로 표현됩니다.

```java
public interface TransactionDefinition {

    int getPropagationBehavior();

    int getIsolationLevel();

    int getTimeout();

    boolean isReadOnly();
}
```


### 1) 트랜잭션 전파

트랜잭션 전파는 이미 실행 중인 트랜잭션이 있을 때 현재 메서드가 어떻게 동작할지를 정의합니다.

대표적인 전파 속성은 다음과 같습니다.

전파 속성 |	설명
-|-
REQUIRED	| 기존 트랜잭션이 있으면 참여, 없으면 생성
REQUIRES_NEW	| 항상 새로운 트랜잭션 생성
SUPPORTS	| 트랜잭션이 있으면 참여, 없으면 없이 실행
NOT_SUPPORTED | 기존 트랜잭션이 있으면 잠깐 멈추고, 트랜잭션 없이 실행



예를 들어 다음과 같이 설정할 수 있습니다.

```java
@Service
public class UserService {

    private final UserLevelService userLevelService;

    public UserService(UserLevelService userLevelService) {
        this.userLevelService = userLevelService;
    }

    @Transactional
    public void upgradeLevels() {
        userDao.getAll();
        userLevelService.updateUsers();
    }
}
```

```java
@Service
public class UserLevelService {

    @Transactional
    public void updateUsers() {
        userDao.update();
    }
}
```


만약 `upgradeLevels()`가 트랜잭션 안에서 실행된다면 `updateUsers()`는 같은 트랜잭션에 참여하게 됩니다.

이는 스프링의 기본 전파 속성이 **REQUIRED**이기 때문입니다.

REQUIRED는 기존 트랜잭션이 존재하면 해당 트랜잭션에 참여하고, 없다면 새로운 트랜잭션을 생성합니다.

이처럼 **메서드 간 호출 관계에서 트랜잭션을 어떻게 공유할지** 결정하는 것이 전파 속성입니다.


### 2) 격리 수준

격리 수준은 여러 트랜잭션이 동시에 실행될 때 데이터 접근을 어떻게 격리할지를 정의합니다.

#### 대표적인 트랜잭션 격리 수준

트랜잭션 격리 수준은 동시에 실행되는 트랜잭션 간 데이터 접근을 얼마나 격리할지를 결정한다.

격리 수준 |	설명	| 방지되는 문제
-|-|-
READ_UNCOMMITTED	| 다른 트랜잭션이 아직 커밋하지 않은 데이터도 읽을 수 있음	| 없음
READ_COMMITTED	| 커밋된 데이터만 읽을 수 있음	| Dirty Read 방지
REPEATABLE_READ	| 같은 트랜잭션 내에서 동일한 조회 결과 보장	| Dirty Read, Non-repeatable Read 방지
SERIALIZABLE	| 트랜잭션을 순차적으로 실행한 것처럼 동작	| Dirty Read, Non-repeatable Read, Phantom Read 방지

#### 간단한 문제 유형
문제 |	설명
-|-
Dirty Read |	다른 트랜잭션이 커밋하지 않은 데이터를 읽는 문제
Non-repeatable Read	| 같은 데이터를 두 번 조회했는데 값이 바뀌는 문제
Phantom Read |	같은 조건으로 조회했는데 새로운 행이 나타나는 문제

예를 들어 다음과 같이 설정할 수 있습니다.

```java
@Transactional(isolation = Isolation.READ_COMMITTED)
public List<User> getUsers() {
    return userDao.getAll();
}
```

격리 수준이 높을수록 데이터 일관성은 높아지지만 동시성은 낮아질 수 있습니다.


### 3) 제한 시간

제한 시간은 트랜잭션이 허용되는 최대 실행 시간입니다.

```java
@Transactional(timeout = 10)
public void batchProcess() {
    // 10초 안에 완료되어야 하는 작업
}
```

지정된 시간이 초과되면 트랜잭션은 롤백됩니다.

### 4) 읽기 전용

읽기 전용 속성은 데이터 변경 없이 조회만 수행하는 트랜잭션임을 명시합니다.

```java
@Transactional(readOnly = true)
public List<User> getUsers() {
    return userDao.getAll();
}
```

이 설정은 ORM이나 데이터베이스가 내부적으로 최적화를 수행할 수 있도록 도와줍니다.

## 6.6.2 트랜잭션 인터셉터와 트랜잭션 속성

스프링에서는 트랜잭션을 직접 코드로 관리하지 않습니다.

대신 **AOP 기반 인터셉터를 통해 트랜잭션을 자동으로 처리합니다.**


## TransactionInterceptor

`TransactionInterceptor`는 메서드 호출을 가로채서 트랜잭션을 관리하는 핵심 컴포넌트입니다.

동작 구조는 다음과 같습니다.

```
Client
   ↓
Proxy
   ↓
TransactionInterceptor
   ↓
Target Object
```

메서드 실행 전
- 트랜잭션 시작

메서드 실행 후
- 정상 종료 → commit
- 예외 발생 → rollback

### 메소드 이름 패턴을 이용한 트랜잭션 속성 지정

스프링에서는 메서드 이름 패턴을 사용해 트랜잭션 속성을 지정할 수 있습니다.

예를 들어 다음과 같은 설정이 가능합니다.

```xml
<tx:advice id="txAdvice">
    <tx:attributes>
        <tx:method name="get*" read-only="true"/>
        <tx:method name="update*"/>
        <tx:method name="delete*"/>
        <tx:method name="*"/>   
    </tx:attributes>
</tx:advice>
```

이 설정은 다음과 같은 의미를 가집니다.

- get* → 읽기 전용 트랜잭션
- update* → 일반 트랜잭션

이 방식은 **공통된 메서드 규칙을 활용해 트랜잭션 정책을 간결하게 정의할 수 있습니다.**


### tx 네임스페이스 설정

스프링에서는 tx 네임스페이스를 이용해 트랜잭션 설정을 선언적으로 작성할 수 있습니다.

**tx 네임스페이스(namespace)** 는 스프링 XML 설정에서 트랜잭션 관련 설정을 쉽게 작성할 수 있게 만든 XML 확장 문법입니다.

원래 스프링 XML은 이렇게 길게 설정해야 했습니다.

```xml
<bean id="txAdvice"
class="org.springframework.transaction.interceptor.TransactionInterceptor">
    <property name="transactionManager" ref="transactionManager"/>
    <property name="transactionAttributes">
        <props>
            <prop key="get*">PROPAGATION_REQUIRED,readOnly</prop>
            <prop key="*">PROPAGATION_REQUIRED</prop>
        </props>
    </property>
</bean>
```

이건 `TransactionInterceptor`를 직접 bean으로 설정하는 방식입니다.

굉장히 장황하기 때문에 스프링이 더 간단한 XML 문법을 제공했습니다.

네임스페이스란 XML에서 특정 기능 그룹을 묶는 문법입니다.

위를 tx 네임스페이스 (`<tx:...>`)를 사용하면 이렇게 줄일 수 있습니다.

```xml
<tx:advice id="txAdvice" transaction-manager="transactionManager">
    <tx:attributes>
        <tx:method name="get*" read-only="true"/>
        <tx:method name="*" propagation="REQUIRED"/>
    </tx:attributes>
</tx:advice>
```

`tx:advice`는 실제로 새로운 기능이 아니라 내부적으로는 **TransactionInterceptor bean을 생성**합니다.


## 6.6.3 포인트컷과 트랜잭션 속성 적용 전략

트랜잭션을 어디에 적용할 것인지는 AOP 포인트컷을 통해 결정됩니다.

### 타입 패턴이나 빈 이름을 이용한 포인트컷

일반적으로 서비스 계층 전체에 트랜잭션을 적용합니다.

```xml
<aop:pointcut id="servicePointcut" expression="execution(* com.example.service..*(..))"/>
```

이 설정은 서비스 패키지의 모든 메서드에 트랜잭션을 적용합니다.

### 메소드 이름 규칙을 이용한 트랜잭션 설정

공통된 메서드 이름 규칙을 사용하면 최소한의 설정으로 트랜잭션 정책을 정의할 수 있습니다.

```
get*   → readOnly
update* → 일반 트랜잭션
```

이 방식은 설정을 단순하게 유지하는 데 도움이 됩니다.

### 프록시 기반 AOP의 한계

스프링 AOP는 프록시 기반으로 동작합니다.

따라서 같은 클래스 내부에서 메서드를 호출하면 트랜잭션이 적용되지 않습니다.

```java
public void methodA() {
    this.methodB(); // 프록시를 거치지 않음
}
```

이 경우 `methodB()`에 트랜잭션이 설정되어 있어도 적용되지 않는다.


## 6.6.4 트랜잭션 속성 적용

트랜잭션 속성을 실제 애플리케이션에 적용할 때는 다음 전략을 사용합니다.

### 트랜잭션 경계 설정의 일원화

트랜잭션 경계는 보통 서비스 계층에서 관리합니다.

이유:
- 여러 DAO 호출을 포함
- 비즈니스 로직의 단위가 서비스 계층

### 서비스 빈에 적용되는 포인트컷 등록

서비스 계층 전체에 트랜잭션을 적용합니다.

```
<aop:advisor advice-ref="txAdvice" pointcut-ref="servicePointcut"/>
```


### 트랜잭션 어드바이스 등록

트랜잭션 속성을 정의한 어드바이스를 등록합니다.

```xml
<tx:advice id="txAdvice">
    <tx:attributes>
        <tx:method name="*" propagation="REQUIRED"/>
    </tx:attributes>
</tx:advice>
```

### 트랜잭션 속성 테스트

트랜잭션 설정은 XML 또는 AOP 설정으로 적용되기 때문에 컴파일 시점에 오류가 드러나지 않습니다. 

따라서 트랜잭션 속성이 의도대로 동작하는지 테스트를 통해 검증하는 것이 중요합니다. 

예를 들어 readOnly 트랜잭션에서 쓰기 작업이 수행되면 예외가 발생하는지 확인함으로써 트랜잭션 설정이 정상적으로 적용되었는지 확인할 수 있습니다.

```java
public class UserService {

    public void getUsers() {
        userDao.update(user); // 일부러 쓰기 작업
    }

}
```

```java
@Test
void readOnlyTransactionAttribute() {
    assertThrows(TransientDataAccessResourceException.class,
            () -> userService.getUsers());
}
```
