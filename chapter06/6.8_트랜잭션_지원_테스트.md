# 6.8 트랜잭션 지원 테스트

## 6.8.1 선언적 트랜잭션과 트랜잭션 전파 속성

**Spring 트랜잭션 전파 속성 정리**

| 전파 속성 | 설명 |
|---|---|
| `REQUIRED` (기본값) | 이미 시작된 트랜잭션이 있으면 참여하고, 없으면 새로 시작 |
| `REQUIRES_NEW` | 항상 새로운 트랜잭션을 시작. 기존 트랜잭션은 보류 |
| `SUPPORTS` | 이미 시작된 트랜잭션이 있으면 참여하고, 없으면 트랜잭션 없이 실행 |
| `NOT_SUPPORTED` | 트랜잭션 없이 실행. 기존 트랜잭션이 있으면 보류 |
| `MANDATORY` | 이미 시작된 트랜잭션이 있어야만 실행. 없으면 예외 발생 |
| `NEVER` | 트랜잭션 없이 실행. 기존 트랜잭션이 있으면 예외 발생 |
| `NESTED` | 이미 시작된 트랜잭션이 있으면 중첩 트랜잭션(savepoint) 시작 |

- DB 트랜잭션은 **작업 단위와 일치**해야 한다. 메소드 실행 전에 트랜잭션이 시작되고, 메소드를 빠져나오면 종료되는 것이 원칙이다.
- **예시**: 사용자 이벤트 신청 처리
  - `process()` → 이벤트 신청자 목록을 전달받아 처리
  - `add()` → `process()` 내부에서 실행되며, 독자적 트랜잭션을 실행하거나 `process()`의 트랜잭션에 **참여**할 수 있다
- **핵심**: 스프링은 트랜잭션 전파 속성을 **선언적**으로 제공하므로, 메소드마다 트랜잭션을 직접 작성할 필요가 없다

## 6.8.2 트랜잭션 동기화 테스트

### 트랜잭션 매니저와 트랜잭션 동기화

- 트랜잭션 동기화는 **트랜잭션 전파**를 위해 중요한 역할을 한다
- `REQUIRED` 전파 속성: 이미 시작된 트랜잭션이 있으면 그 트랜잭션에 참여
- `@Autowired`로 `PlatformTransactionManager`를 주입받아 테스트 가능

**개별 트랜잭션 실행 (각각 별도 트랜잭션)**

```java
// 각 메소드 호출마다 독립적인 트랜잭션이 생성되고 종료된다
userService.deleteAll();  // 트랜잭션 1: 시작 → 커밋
userService.add(user1);   // 트랜잭션 2: 시작 → 커밋
userService.add(user2);   // 트랜잭션 3: 시작 → 커밋
```

**하나의 트랜잭션으로 통합**

```java
// 테스트 코드에서 트랜잭션을 미리 시작하여 하나로 묶는다
DefaultTransactionDefinition txDefinition = new DefaultTransactionDefinition();
TransactionStatus txStatus = transactionManager.getTransaction(txDefinition);

userService.deleteAll();  // 기존 트랜잭션에 참여 (REQUIRED)
userService.add(user1);   // 기존 트랜잭션에 참여
userService.add(user2);   // 기존 트랜잭션에 참여

transactionManager.commit(txStatus);
```

### 트랜잭션 동기화 검증

- 트랜잭션 매니저로 **readOnly 트랜잭션**을 생성한 뒤 실행
- `deleteAll()` 호출 시 **예외 발생** → 동기화가 제대로 동작함을 확인

### 롤백 테스트

- 테스트 코드에서 트랜잭션을 제어하여 **롤백을 적용**하는 기법

```java
DefaultTransactionDefinition txDefinition = new DefaultTransactionDefinition();
TransactionStatus txStatus = transactionManager.getTransaction(txDefinition);

try {
    userService.deleteAll();
    userService.add(user1);
    userService.add(user2);
} finally {
    transactionManager.rollback(txStatus);  // 테스트 후 항상 롤백
}
```

> 개인 의견: 테스트 코드에 상위 트랜잭션으로 전파시키는 게 맞는지는 의문

## 6.8.3 테스트를 위한 트랜잭션 애노테이션

| 애노테이션 | 설명 |
|---|---|
| `@Transactional` | 테스트에도 적용 가능. 트랜잭션 경계가 자동 설정되어 테스트 내 작업을 하나로 묶어줌 |
| `@Rollback` | 테스트 종료 후 **자동 롤백**. DB에 실제 커밋되지 않음. `@Rollback(false)`로 커밋 가능 |
| `@NotTransactional` / `Propagation.NEVER` | 트랜잭션 없이 실행 |

### 효과적인 DB 테스트

- 테스트 코드를 **슬림하게** 작성하라는 것이 핵심

> 개인 의견: 현실에서 통합 테스트를 슬림하게 작성하는 게 가능한가...?는 의문
